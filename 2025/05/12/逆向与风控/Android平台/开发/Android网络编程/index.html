Android网络编程简单了解

概述

在Android上进行网络通信开发是指利用Android平台提供的网络能力实现应用程序与远程服务器或其他设备之间的数据交换，其核心是通过各种网络协议和技术建立连接、传输数据并处理响应。

Android网络通信主要分为基于HTTP/HTTPS协议的高层通信（如HttpURLConnection、OkHttp、Retrofit等框架）和基于TCP/UDP协议的底层Socket通信两类：

  • 前者适用于标准的Web服务交互。
  • 后者则用于需要自定义协议或实时性要求高的场景（如即时通讯、游戏联机）。

开发时需遵循Android的网络线程模型（主线程禁止网络操作）、权限声明（INTERNET权限）及安全规范（HTTPS加密、数据校验），同时要考虑设备网络状态变化（ ConnectivityManager 监听）、后台执行限制以及电量优化等系统特性。

现代Android开发通常结合协程（Coroutine）或RxJava等异步编程模型来简化网络请求的线程管理和响应处理。

开门见山

毫无疑问的最原始的是基于TCP/UDP协议的原始的套接字网络通信(Socket模型 - The Peak Tower)，虽然能够实现任意自定义协议的设备间通信，并提供最基础的网络数据传输能力。

[客户端] --(字节流)--> [Socket] --(字节流)--> [服务端]

但还是有以下缺点

  • 需手动处理线程、连接池、协议解析
  • 易出现内存泄漏和性能问题

在Android 1.0+，引入了Java标准库的HTTP客户端【HttpURLConnection（系统API）】，解决了：1.简化HTTP协议实现（无需手动拼装请求头）。2.支持HTTPS和连接复用。

[App] --HTTP请求--> [HttpURLConnection] --解析响应--> [JSON/XML]

但还是存在如下问题：

  • API设计繁琐（需手动处理InputStream）
  • 不同版本实现不一致（Android 2.3前有Bug）

时间来到2014年，由Square公司推出OkHttp（现代HTTP客户端），这是一个高效的HTTP/2客户端。

[App] --> [OkHttp] --> [拦截器] --> [缓存/压缩/重试] --> [服务器]

成功实现了：

  • 连接池复用降低延迟
  • 透明的GZIP压缩
  • 自动重试和故障转移
  • 拦截器链支持（日志、缓存等）

由于前后端分离以及RESTFul接口规范的流行，在到2013年Square基于OkHttp推出的Retrofit（类型安全封装），其核心在于RESTful API的声明式客户端。

[App Interface] --注解--> [Retrofit] --转换--> [OkHttp] --网络请求--> [服务器]

解决了：

  • 将HTTP API转为Java接口（通过注解配置）
  • 自动序列化（JSON↔Object）
  • 完美结合RxJava/Coroutine

回顾整个发展过程：

  • 实线箭头表示技术演进主线
  • 虚线箭头表示跨代解决的问题

对比

技术 核心贡献 解决的问题 遗留痛点 典型场景
Socket 原始TCP/UDP通信 自定义协议、实时数据传输 需手动处理所有底层细节 物联网、游戏联机
HttpURLConnection 系统级HTTP封装 简化HTTP请求、支持HTTPS API冗余、版本兼容性问题 简单HTTP请求
OkHttp 高性能HTTP/2客户端 连接复用、拦截器、自动压缩/重试 仍需手动解析响应数据 需要精细控制的网络请求
Retrofit 声明式REST客户端 将API转换为接口、自动序列化 依赖OkHttp，灵活性降低 现代RESTful API交互

OkHttp通信

使用前注意Gradle 依赖配置

  • // OkHttp (最低要求4.x)
    implementation(“com.squareup.okhttp3:okhttp:4.11.0”)
  • // Retrofit 核心库
    implementation(“com.squareup.retrofit2:retrofit:2.9.0”)
  • // 数据转换器（按需选择）
    implementation(“com.squareup.retrofit2:converter-gson:2.9.0”) // JSON转换
    implementation(“com.squareup.retrofit2:converter-scalars:2.9.0”) // 原始字符串转换
  • // 异步适配器（按需选择）
    implementation(“com.squareup.retrofit2:adapter-rxjava3:2.9.0”) // RxJava3支持

并在AndroidManifest.xml添加：

<uses-permission android:name="android.permission.INTERNET" />

OkHttp是基于Http协议封装的一套请求客户端，虽然它也可以开线程，但根本上它更偏向真正的请求，跟HttpClient,HttpUrlConnection的职责是一样的。
OkHttp主要负责socket部分的优化，比如多路复用，buffer缓存，数据压缩等等。

如下是OkHttp所提供的常用功能：

功能模块 解决的问题 实现方式 传统方案对比
连接池管理 TCP握手开销大、连接复用率低 维护透明连接池（默认支持5个并发，5分钟存活） HttpURLConnection需手动配置复用
HTTP/2支持 多路复用、头部压缩提升性能 自动协商HTTP/2或SPDY 仅HTTP/1.1，线头阻塞问题严重
拦截器链 统一处理日志、缓存、重试等横切关注点 责任链模式（可自定义Interceptor） 需在每个请求中重复编码
自动压缩 带宽浪费 自动添加Accept-Encoding: gzip头 需手动处理压缩/解压
缓存控制 重复请求网络资源 遵循HTTP缓存规范（Cache-Control） 需依赖外部缓存库
超时与重试 弱网环境下请求失败 默认配置： • 连接超时：10秒 • 读取超时：10秒 • 写入超时：10秒 需手动实现超时回调
HTTPS安全 中间人攻击、证书校验 默认启用证书链验证 支持证书锁定（Certificate Pinning） 需自行配置SSLContext
同步/异步统一 线程管理复杂 enqueue()自动切换线程 execute()同步执行 AsyncTask/ThreadPool易内存泄漏
WebSocket 实时双向通信 内置WebSocket协议支持

Retrofit通信

准确来说，Retrofit 是一个 RESTful 的 HTTP 网络请求框架的封装。

因为网络请求的工作本质上是 OkHttp 完成，而 Retrofit 仅负责网络请求接口的封装。

  • App应用程序通过Retrofit请求网络，实际上是使用Retrofit接口层封装请求参数、Header、Url 等信息，之后由 OkHttp 完成后续的请求操作。

  • 在服务端返回数据之后，OkHttp 将原始的结果交给 Retrofit，Retrofit根据用户的需求对结果进行解析。
