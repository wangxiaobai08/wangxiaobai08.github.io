Frida-Core源码浅析

先来看看Frida的基本使用：

  1. 下载Frida的Server端->frida-server-15.1.4-android-arm64.xz，下载完成之后解压。然后push到手机的data/local/tmp目录下

    adb push .\frida-server-15.1.4-android-arm64 /data/local/tmp/frida

  2. 然后修改权限：chmod 777 frida

  3. 直接运行frida服务：./frida

  4. 开启端口转发：

    adb forward tcp:27042 tcp:27042
    adb forward tcp:27043 tcp:27043

  5. PC端输入：frida-ps -U

frida-server

frida-server 是 Frida 工具链的核心组件，主要用于在 目标设备（通常是 Android 或 Linux） 上运行，以便与 Frida 客户端（如 frida-tools、frida-python）通信。

frida-server 的核心作用

提供调试和注入能力

  • Frida 的核心功能（如动态 Hook、内存读写、调用函数）需要直接与目标进程交互。
  • frida-server 作为守护进程，运行在目标设备上，负责执行 Frida 客户端的指令（如注入 JavaScript、修改内存）。

绕过权限限制

  • 普通应用无法直接 ptrace 其他进程（Android 的 ptrace_scope 限制）。
  • frida-server 以 root 或高权限运行，可以绕过这些限制，实现跨进程调试和注入。

支持网络通信

  • 默认情况下，Frida 通过 TCP 与客户端通信（如 frida -H 192.168.x.x）。
  • frida-server 监听端口（默认 27042），允许远程设备（如 PC）通过 USB 或网络连接。

在这里提前了解一下 D-Bus

D-Bus

D-Bus 是 Linux 桌面环境的 标准 IPC 机制，允许不同进程（甚至是不同语言编写的工具）通过统一的接口通信。我们知道最初Linux是不支持桌面环境的。

D-Bus 最初是为 GNOME/KDE 等桌面环境 设计的，核心目标是解决桌面组件（如窗口管理器、音量控制、网络设置）之间的通信需求。

典型用例：

  • 用户点击音量按钮 → 通过 D-Bus 通知音频服务调整音量。
  • 文件管理器需要挂载磁盘 → 通过 D-Bus 调用 udisks2 服务。

与普通 Linux IPC 的对比

特性 D-Bus（桌面环境） 普通 Linux IPC
主要用途 桌面组件通信、用户交互 系统级进程通信（如 daemon、高性能服务）
通信模式 基于消息（异步/同步） 多样化（管道、消息队列、共享内存、Socket）
抽象层级 高级（对象/接口/方法） 低级（字节流、信号、文件描述符）
服务发现 支持（通过总线名称） 需手动实现（如配置文件或固定路径）
性能 较高延迟（适合低频操作） 极高（适合高频数据传输）
典型用户 GUI 应用、系统服务（如 NetworkManager） 后台服务、嵌入式系统、内核模块

Frida 在 Linux 桌面环境 下使用 D-Bus 的原因：

与桌面工具集成

  • 场景：Frida 的 GUI 调试器（如 frida-gui）需要与桌面环境交互（如弹出对话框、通知状态）。
  • 实现：通过 D-Bus 调用 org.freedesktop.Notifications 服务显示
  • 优势：D-Bus 的接口描述语言（如 .service 文件）允许 Frida 暴露统一的 HostSession API，供不同语言（Python、C、JS）调用。
    • 对比：若用自定义协议，需为每种语言实现客户端库。

Frida 通过 D-Bus 暴露核心功能（如 HostSession），使得第三方工具（如 GUI 调试器、自动化脚本）无需依赖 Frida 的私有协议，直接通过 D-Bus 调用功能。

在 Linux 桌面环境中，许多系统服务（如网络、硬件管理）本身通过 D-Bus 暴露接口。Frida 使用 D-Bus 可以更方便地与这些服务交互（例如通过脚本监控网络状态）

  • 例如：一个 Python 脚本可以通过 dbus-python 库直接调用 Frida 的 attach() 方法。

同时在Frida使用D-Bus 的接口（如 HostSession、TransportBroker）强制将功能模块化，每个接口对应一组明确的操作。例如：

  • HostSession：进程管理（附加、启动、枚举）。
  • TransportBroker：文件传输。
  • AuthenticationService：认证逻辑。
    这种分离使得代码更易维护和扩展。

主要Frida通信不全是D-Bus:

场景 协议选择 原因
客户端 ↔ frida-server（Linux） D-Bus 方便与桌面工具集成，支持多语言调用
frida-server ↔ frida-agent 共享内存/Unix Socket 高频操作需低延迟，D-Bus 序列化开销过大

两种常见使用模式

我们知道Frida使用有两种模式：

其中**spawn() + resume()** 是 注入模式 的核心操作，适合控制进程从启动开始的行为。而附加模式 直接操作运行中的进程，无需 spawn/resume。两者底层均依赖 ptrace，但应用场景和权限要求不同。

操作 注入模式（Spawn） 附加模式（Attach）
进程状态 从零启动，初始挂起 附加到已运行的进程
API 调用 需调用 spawn() + resume() 直接调用 attach()
适用场景 需控制进程从启动开始的行为 动态修改已运行进程
权限要求 需能启动新进程（可能需 root） 需 ptrace 权限（可能需 root）

Q1: 为什么需要 spawn() 后挂起？

挂起进程后，Frida 可以在主线程执行前注入脚本，确保 Hook 所有关键函数（如 JNI_OnLoad）。

Q2: resume() 后还能继续注入吗？

可以，但动态注入可能错过早期代码（如构造函数）。推荐在 resume() 前完成核心 Hook。

Q3: 附加模式是否需要 resume()？

不需要，附加模式不会暂停目标进程（除非手动调用 Process.suspend()）。

总结

attach模式

attach到已经存在的进程，核心原理是ptrace修改进程内存，如果进程处于调试状态（traceid不等于0），则attach失败。

frida 的进程注入是通过 ptrace 实现的，注入之后会在远端进程分配一段内存将 agent 拷贝过去并在目标进程中执行代码，执行完成后会 detach 目标进程。

这也是为什么在 frida 先连接上目标进程后还可以用 gdb 等调试器连接，而先 gdb 连接进程后 frida 就无法再次连上的原因。

frida-agent 注入到目标进程并启动后会启动一个新进程与 host 进行通信，从而 host 可以给目标进行发送命令，比如执行代码，激活/关闭 hook，同时也能接收到目标进程的执行返回以及异步事件信息等。

spawn模式

启动一个新的进程并挂起，在启动的同时注入frida代码，适用于在进程启动前的一些hook，如hook RegisterNative等，注入完成后调用resume恢复进程。

frida-helper

在前面是spawn和attach两种模式中，都提到了frida-agant和frida-helper这些到底是什么呢？

frida-agant 在大多数时候充当了 app 内部的服务端。在用户向应用传递新脚本的时候，通过 RPC 服务与 frida-agant 通信，由它来负责注入 hook 。

frida-helper 是 frida-server 的辅助模块，专门处理 跨进程内存操作、Shellcode 执行、动态链接器交互 等脏活累活。

这三者关系可以用「医院手术团队」 的比喻来理解 Frida 的各个组件：

组件 比喻角色 功能
frida-server 主刀医生 在目标设备（如手机）上运行，拥有最高权限，负责调度所有操作（如开刀、注射）。
frida-agent 手术助手 注入到目标 App 内部，直接操作“病人器官”（Hook 函数、修改内存）。必须在目标 App 内部运行，否则无法直接修改其内存。
frida-helper 麻醉师/护士 辅助 frida-server，处理低级任务（如内存扫描、线程管理）。分担底层任务，让 frida-agent 更轻量（类似护士让医生专注关键步骤）。

可以同Window下DLL注入类比，**frida-agent**就像是被被注入到目标进程的dll文件。

bootstrap

引导代码是private async BootstrapResult bootstrap (size_t loader_size, Cancellable? cancellable)

负责在目标进程中 加载并初始化 Frida 的运行时环境（frida-agent）。是Frida 注入引擎的核心实现部分，整体流程如下：

(1) 准备引导代码

bootstrapper_code 是一段编译好的机器码（类似 Shellcode），用于在目标进程内部：

  • 分配内存。
  • 加载 frida-agent.so。
  • 初始化 Frida 的通信管道。

(2) 内存分配策略

优先复用目标进程的 mmap：若目标进程的 libc 与注入端一致，直接调用其 mmap 分配内存。备用方案（无 mmap）：通过 RemoteCallBuilder 执行自定义分配逻辑（见下文）。

(3) 远程调用

在目标进程的上下文中执行 bootstrapper_code。

关键操作：

  • 在目标进程的栈上预留空间（用于传递参数）。
  • 通过 bootstrap_ctx 结构体传递配置（如内存地址、页大小）。
  • 调用 bootstrapper_code 并检查返回状态。

可以看到frida-helper组件的核心目标是在目标进程中搭建 Frida 的运行环境（类似“寄生”）：

  • 动态分配内存：在目标进程中开辟空间，用于加载 Frida 的引导代码（bootstrapper）。
  • 执行远程调用：让目标进程主动加载 frida-agent，完成 Hook 环境的初始化。

frida-agent

frida-agent 是 Frida 的核心组件之一，负责 注入到目标进程 并执行动态插桩（Dynamic Instrumentation）任务。在 注入模式 下，通过 frida-server与客户端通信。它的主要功能包括：

功能分类 具体作用 相关 API/技术
执行 JavaScript 或 Native 代码 - 运行注入的 JavaScript 脚本（Hook、内存读写等） - 调用目标进程的 Native 函数 Script.load() NativeFunction Memory.read/write()
动态插桩（Hooking） - 拦截函数调用（C/C++/ObjC/Java） - 修改函数参数、返回值或寄存器 Interceptor.attach/detach() Interceptor.replace()
进程内通信 - 与 Frida 客户端（Python/Node.js）交互 - 通过 IPC 与 frida-server/frida-gadget 通信 frida-tools frida-python D-Bus（Linux） TCP/Unix Socket（嵌入式模式）
管理资源 - 脚本生命周期管理（加载/卸载） - 线程安全，防止目标进程崩溃 Script.unload() Thread.safe 相关机制
