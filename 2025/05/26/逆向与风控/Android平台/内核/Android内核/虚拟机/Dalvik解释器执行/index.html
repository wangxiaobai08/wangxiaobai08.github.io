Dalvik解释器执行流程

解释指令前的准备工作

从外部进入解释器的调用链如下：

dvmCallMethod -> dvmCallMethodV -> dvmInterpret这三个函数是在解释器取指令，选分支之前被调用，主要负责一些准备工作，包括分配虚拟寄存器，放入参数，初始化解释器参数等。其中dvmCallMethod，直接调用了dvmCallMethodV.下面分析下后两个函数。

dvmCallMethodV

dvmCallMethodV 是 Dalvik 虚拟机中用于调用方法的核心函数之一，它负责准备方法调用环境并将控制权转移给目c标方法（无论是本地方法还是 Java 方法）。

函数参数说明

  • self: 当前线程指针
  • method: 要调用的方法对象
  • obj: 对于实例方法，这是”this”对象；对于静态方法，可以为NULL
  • fromJni: 指示调用是否来自JNI
  • pResult: 用于存储方法返回值的指针
  • args: 可变参数列表，包含方法的参数

主要执行流程

1. 获取方法签名

method->shorty 是方法的简短签名，第一个字符是返回类型，后面的字符是参数类型,例如 onCreate 方法的签名可能是 VL，表示返回void，参数为对象

2. 准备调用环境

callPrep 函数完成以下工作：

  • 访问权限检查
  • 分配函数调用栈帧
  • 在栈中维护虚拟寄存器列表
  • 返回方法所属的类对象
3. 设置参数寄存器指针

ins 指针指向第一个参数的位置,Dalvik 使用基于寄存器的架构，参数和局部变量都存储在虚拟寄存器中

4. 处理”this”指针

对于非静态方法，第一个参数是”this”指针,将”this”对象放入第一个虚拟寄存器。verifyCount 用于跟踪已处理的参数数量

5. 处理其他参数

根据方法签名中的参数类型描述符，逐个处理参数：

处理64位类型（double/long）

使用memcpy将64位值拷贝到两个连续的32位寄存器中。指针前进2个位置（每个位置32位）

处理float类型

JNI规范中float被提升为double传递，这里需要转换回float。使用dvmFloatToU4进行浮点数到32位表示的转换

处理对象引用

对于来自JNI的调用，需要解码间接引用，本地调用直接使用对象引用

处理其他基本类型

包括boolean、byte、char、short、int等，都作为32位值传递

6. 执行方法
本地方法执行

直接调用方法关联的本地函数指针，参数包括：栈帧指针、结果指针、方法对象和线程对象

Java方法解释执行

进入解释器执行字节码，dvmInterpret 是Dalvik的解释器入口

7. 清理栈帧

方法执行完成后弹出栈帧，恢复调用者的执行环境

可以看到dvmCallMethodV主要进行了：

  1. 寄存器分配：Dalvik基于寄存器架构，dvmCallMethodV负责将参数放入正确的虚拟寄存器中。

  2. 参数处理：根据方法签名动态处理不同类型的参数，包括基本类型和对象引用。

  3. 本地方法与Java方法：使用不同的执行路径，本地方法直接调用函数指针，Java方法进入解释器。

  4. 栈帧管理：通过callPrep和dvmPopFrame管理调用栈，确保方法调用的隔离性。

  5. JNI支持：处理来自JNI的特殊情况，如间接引用解码和参数类型转换。

dvmInterpret

dvmInterpret 是 Dalvik 虚拟机的核心解释器入口函数，负责初始化执行环境并选择适当的解释器来执行 Java 字节码。

函数原型

  • self: 当前线程对象指针
  • method: 要执行的 Java 方法对象
  • pResult: 用于存储方法返回值的指针

执行流程详解

1. 保存当前解释器状态

创建局部变量保存当前解释器状态 (interpSaveState)，将当前状态保存到线程状态中，形成链表结构（支持嵌套解释器调用）

保存当前的执行子模式 (savedSubModes)，用于异常处理等特殊情况

2. JIT 相关初始化

如果启用了 JIT 编译，保存调用者需要保护的寄存器，calleeSave 数组用于保存浮点寄存器状态。这是 JIT 与解释器交互的重要机制

3. 调试支持

跟踪引用检查的初始化，标记当前为方法入口，用于调试目的

4. JIT 状态初始化

初始化 JIT 状态为 kJitNot（未使用 JIT），这个状态会在执行过程中可能被修改

5. 解释器环境初始化

设置要执行的方法 (method)，初始化当前栈帧指针 (curFrame)，设置程序计数器 (pc) 指向方法的指令开始处 (method->insns)

6. 方法合法性检查

断言确保不是本地方法（本地方法不应进入解释器）

检查方法所属类的状态：必须至少正在初始化 (CLASS_INITIALIZING)，不能是错误状态 (CLASS_ERROR)

如果检查失败，打印错误信息并终止虚拟机

7. 选择解释器实现

定义解释器函数指针类型

根据全局执行模式 (gDvm.executionMode) 选择解释器：

  • kExecutionModeInterpFast: 使用快速解释器 (dvmMterpStd)
  • JIT 相关模式也使用快速解释器
  • 默认使用便携式解释器 (dvmInterpretPortable)
8. 调用解释器

通过函数指针调用选定的解释器，传入当前线程对象作为参数

9. 保存返回值

将解释器执行结果保存到输出参数中

10. 恢复解释器状态

恢复之前的解释器状态，如果有 JIT，恢复调用者保存的寄存器，恢复之前保存的执行子模式。

解释器流程分析

Thread Code

问题：android - How does mterp (Dalvik VM) organize its byte-code interprete loop? - Stack Overflow

实现解释器的一个常见思路如下代码，循环取指令，然后判断指令类型，去相应分支执行，执行完成后，再返回到switch执行下条指令。

但是当每次执行一条指令，都需要重新判断下条指令类型，然后选择switch分支，这是个昂贵的开销。Dalvik为了解决这个问题，引入了Thread Code技术。

简单的说就是在执行函数之前，建立一个分发表GOTO_TABLE，每条指令在表中有一个对应条目，条目里存放的就是处理该条指令的handler地址。

比如invoke-super指令,它的opcode为6f，那么处理该条指令的handler地址就是：GOTO_TABLE[6f].那么在每条指令的解释程序末尾，都可以加上取指动作，然后goto到下条指令的handler。

dvmInterpretPortable

dvmInterpretPortable 是 Dalvik 虚拟机中便携式解释器的核心实现

函数签名和基本结构

  • self: 当前线程的 Thread 对象指针
  • 无返回值，结果通过线程的 interpSave.retval 传递

分发表初始化

这是一个宏定义的跳转表，为每条 Dalvik 指令生成对应的处理标签，实际展开后类似于一个大的 switch-case 结构c，但使用 goto 实现更高效率的跳转

执行环境初始化：

从线程状态中初始化解释器的核心变量：

  • 当前方法
  • 程序计数器
  • 栈帧指针
  • 返回值
  • 方法所属类的 DEX 信息

在调试模式下记录方法进入c事件，设置方法入口标志

方法调用初始化：

初始化方法调用变量，-1 表示c没有待调用的方法。

启动解释执行

FINISH 是一个关键宏，负责获取下一条指令并跳转到对应的处理程序。参数 0 表示从当前 pc 开始执行。

指令执行机制
FINISH 宏
  • ADJUST_PC: 调整程序计数器
  • FETCH: 获取指令操作码
  • 通过 handlerTable 跳转到对c应的指令处理程序
指令处理模板

以 NOP 指令为例：

  • HANDLE_OPCODE: 定义指令处理程序的开始
  • FINISH(1): 处理完成后移动到下一条指令
  • OP_END: 标记指令处理程序结束c

典型指令处理示例

加法指令 (OP_ADD_INT)

  1. 解码指令参数
  2. 日志记录
  3. 执行加法操作：从源寄存器取值相加，结果存入目标寄存器
  4. 移动到下一条指令

方法调用指令 (OP_INVOKE_VIRTUAL)

  1. 使用 GOTO_invoke 宏处理虚方法调用
  2. false 表示不使用 range 形式参数
