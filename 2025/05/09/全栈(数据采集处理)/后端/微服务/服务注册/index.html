但随着网站规模的逐渐扩大，微服务之间的关联度日渐复杂：

传统 gRPC 静态注册服务的局限性

1. 新加一个服务

在传统的 gRPC 静态注册模式下，如果想增加一个新的服务实例，需要手动配置该服务实例的IP地址和端口。这种方式有一些显著的局限性：

  • 手动配置的复杂性：每当新增一个服务实例时，你需要手动更新客户端的配置或者服务端的配置。假设服务是跨多台机器部署的，每台机器的IP和端口都需要手动配置到客户端。这种做法显然是非常繁琐的，尤其是在服务规模较大的情况下。
  • IP 地址变化的挑战：服务实例的 IP 地址和端口是动态分配的，尤其是在容器化、云环境或者微服务架构中。每次服务实例新增时，它的 IP 地址可能发生变化，导致客户端无法获取到正确的服务地址。你必须在客户端手动更新这些信息。
  • 缺乏动态性：一旦服务实例启动并被静态配置，客户端无法自动感知服务的变化。如果服务的 IP 或端口发生了变化，客户端需要重新配置，这种静态配置不能适应服务的伸缩、故障恢复等动态场景。

示例： 假设有一个微服务架构，某个服务的实例需要动态增加。当新增实例时，原本的客户端或其他服务可能并不了解新的实例存在，必须手动更新客户端的配置。假如服务实例多且频繁增加，这种方式就会变得难以管理。

2. 并发度过高（添加同一服务的机器）

当有多个服务实例需要处理大量的并发请求时，服务实例的添加与负载均衡会成为一个难题。静态注册的方式在这个场景下也存在显著的局限性：

  • 负载均衡问题：传统的静态注册方式通常会在客户端或者服务端进行负载均衡。这要求客户端提前知道所有的服务实例地址，才能进行请求的分发。然而，随着服务实例的不断增加，维护一个包含所有服务实例的静态配置会变得越来越困难。当并发度过高时，新的服务实例会不断加入，原有的负载均衡机制可能无法实时适应这些变化，导致请求没有被均匀地分发。
  • 更新滞后：当服务实例的数量发生变化时，静态配置无法及时更新，导致客户端或负载均衡器没有及时感知到新的服务实例，进而影响系统的性能。假设多个新的服务实例同时启动，且负载均衡策略未能及时更新，原本的请求可能会集中到部分实例上，导致资源过载，影响系统的可用性和稳定性。
  • 并发请求的处理瓶颈：假设有大量的并发请求需要处理，而系统是基于静态注册的。如果服务的实例数不足以满足并发请求量，那么在高并发的场景下，静态注册无法进行动态扩展和缩放。你必须手动增加服务实例，并更新所有客户端的配置，而当并发量继续增加时，这种手动更新的过程可能导致系统出现瓶颈。

示例： 假设有一个微服务架构中的订单服务，它是一个高并发的服务。每当请求量增加，你需要手动添加新的服务实例，并在客户端手动配置服务实例的地址。这种方式不仅增加了运维的负担，而且每当新增的服务实例无法实时生效时，会导致请求集中到有限的几个服务实例上，影响系统的负载均衡和响应时间。

consul注册中心的引入

Consul 是一个开源的工具，用于服务发现、配置管理和健康检查。它主要用于帮助管理大规模的分布式系统，特别是微服务架构中的服务注册与发现。

从上面可以看到静态注册的许多不便，而consul便是解决这些问题的一种解决方案。

举个直白的例子：想象一下你想去一家餐厅预定座位，这里有两种预定方式，一种是传统的电话预定（对应 gRPC 静态注册），另一种是使用现代化的餐厅预定平台（对应 Consul）。

在传统的电话预定方式中：

  • 餐厅地址和电话是固定的：你必须知道餐厅的准确地址和电话（服务的 IP 地址和端口），然后通过电话确认座位。如果是多个餐厅，每个餐厅有固定的地址和电话，你需要手动记住或者查找这些信息。
  • 服务实例增加复杂：如果有新的餐厅开业，或者某个餐厅增加了新分店，你需要手动更新每个餐厅的电话和地址，确保这些信息能够被客户知道。并且，当餐厅的座位数发生变化时，客户需要重新拨打电话确认，服务端也需要手动更新这些信息。
  • 动态变化无法自动感知：如果某个餐厅因为装修或其他原因暂时关闭，你需要再次拨打电话确认是否有空位，这样的过程增加了很多人工干预。服务的动态变化（如服务实例增加、健康检查等）需要你手动调整。

而在现代的餐厅预定平台中，系统会自动处理所有的预定流程：

  • 服务发现和自动更新：你不需要记住餐厅的地址和电话。所有餐厅的信息（如地址、座位数、健康状况等）都集中在平台中，平台会自动更新。当餐厅新增分店或者变动时，平台会自动更新其信息，客户可以立即查询到最新的餐厅数据。
  • 健康检查与动态调整：平台会自动检查每个餐厅的可用性。如果餐厅的座位数满了，或者因为某种原因暂时关闭，平台会实时调整并排除不可用的餐厅，确保你不会预定到满员或关闭的餐厅。
  • 负载均衡与扩展性：如果你想预定座位，平台会根据实时情况推荐可用的餐厅，并自动分配空闲的座位。如果某个餐厅突然增加了更多座位，平台会及时更新，客户无需关心具体的餐厅地址和座位数变动，系统会自动处理服务实例的负载均衡。

服务的注册与发现

微服务注册与发现类似于生活中的”电话通讯录”的概念，它记录了通讯录服务和电话的映射关系。在分布式架构中，服务会注册进去，当服务需要调用其它服务时，就这里找到服务的地址，进行调用。

步骤如下：

  1. 你先要把”好友某某”记录在通讯录中。
  2. 拨打电话的时候通过通讯录中找到”好友某某”，并拨通回电话。
  3. 当好友某某电话号码更新的时候，需要通知到你，并修改通讯录服务中的号码。

从这个过程中我们看到了一些特点：

1、把 “好友某某” 的电话号码写入通讯录中，统一在通讯录中维护，后续号码变更也是更新到通讯录中，这个过程就是服务注册的过程。

2、后续我们通过”好友某某”就可以定位到通讯录中的电话号码，并拨通电话，这个过程理解为服务发现的过程。

而我们微服务架构中的服务注册与发现结构如下图所示：

图片中是一个典型的微服务架构，这个结构中主要涉及到三大角色：

  • provider - 服务提供者
  • consumer - 服务消费者
  • register center - 注册中心

它们之间的关系大致如下：

  • 每个微服务在启动时，将自己的网络地址等信息（微服务的ServiceName、IP、Port、MetaData等）注册到注册中心，注册中心存储这些数据。
  • 服务消费者从注册中心查询服务提供者的地址，并通过该地址调用服务提供者的接口。
  • 各个微服务与注册中心使用一定机制（例如心跳）通信。如果注册中心与某微服务长时间无法通信，就会注销该实例。

优点如下：

  • 解耦：服务消费者跟服务提供者解耦，各自变化，不互相影响
  • 扩展：服务消费者和服务提供者增加和删除新的服务，对于双方没有任何影响
  • 中介者设计模式：用一个中介对象来封装一系列的对象交互，这是一种多对多关系的中介者模式。

从功能上拆开主要有三块：服务注册、服务发现，和注册中心。我们一个一个来看。

服务发现

服务发现实际就是我们查询已经注册好的服务提供者，比如 p->p.queryService(serviceName)，通过服务名称查询某个服务是否存在，如果存在，返回它的所有实例信息，即一组包含ip 、 port 、metadata元数据信息的endpoints信息。这一组endpoints信息一般会被缓存在本地，如果注册中心挂掉，可保证段时间内依旧可用，这是去中心化的做法。

服务发现的方式一般有两种：

  • 拉取的方式：服务消费方（Consumer）主动向注册中心发起服务查询的请求。
  • 推送的方式：服务订阅/通知变更（下发）：服务消费方（Consumer）主动向注册中心订阅某个服务，当注册中心中该服务信息发生变更时，注册中心主动通知消费者。

拉取与推送方式的比较

特性 拉取方式 推送方式
主动性 消费者主动查询服务信息 注册中心主动通知服务信息变化
查询频率 定期查询，查询频率由消费者控制 注册中心通知时触发更新，实时性强
负载 消费者需要定期发起请求，增加注册中心的负担 减少了对注册中心的频繁查询，降低负载
实现复杂性 简单，易于实现 较为复杂，可能需要使用长连接、消息队列等机制

拉取方式适用于服务信息变化不频繁或服务消费者不需要实时获取服务信息更新的场景，通常比较简单。

推送方式适用于需要实时感知服务信息变化的场景，可以确保服务消费者获取最新的服务地址和实例信息，但实现较为复杂。

健康检查

Consul的健康检查是确保微服务健康运行的重要功能。它通过监控服务实例的健康状态，帮助保证只有健康的服务实例能够接收请求，而不健康的实例会被从负载均衡池中移除，防止客户端调用失败。

健康检查的必要性

假设我们有一个电子商务系统，包含多个微服务，如订单服务、支付服务、库存服务等。这些服务可能部署在多个服务器或容器中，而这些服务可能会在运行过程中出现故障或性能下降。例如，支付服务可能因为数据库连接问题而不可用，或者库存服务由于高并发导致超时。

健康检查的基本思想是让服务提供者定期报告自己的健康状态，然后注册中心根据这些报告来决定是否将服务实例从可用列表中移除。

健康检查的原理

  1. 服务注册时定义健康检查：在服务注册到Consul时，服务提供者可以指定健康检查的配置。健康检查可以是HTTP检查、TCP检查、脚本检查等，Consul会定期执行这些健康检查。
  2. 健康检查定期运行：Consul通过健康检查来定期检测服务实例的可用性。如果健康检查成功（如HTTP返回状态码200），则认为服务健康；如果失败（如HTTP返回状态码500或TCP连接失败），则认为服务不健康。
  3. 健康状态反馈：每次健康检查的结果会更新服务在Consul中的健康状态。健康的服务实例仍然可以接受请求，而不健康的服务实例会从负载均衡池中移除。
