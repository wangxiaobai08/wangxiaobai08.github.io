Docker技术原理浅析

什么是docker？

获取 Docker | Docker 文档

Docker 的定义

Docker 是一个开源的 容器化平台，用于自动化应用程序的部署、扩展和管理。它通过容器技术，允许开发人员将应用及其所有依赖打包成一个标准化的单元（容器），并在任何环境中一致地运行，无论是开发环境、测试环境，还是生产环境。

具体来说，Docker 提供了一个轻量级、可移植、可扩展的容器化解决方案，它通过将应用程序及其依赖打包为一个容器镜像，从而解决了“在我电脑上能跑”的问题，确保了不同环境中的应用行为一致性。

Docker 的核心概念

容器（Container）：是 Docker 中的一个运行时实例，它打包了应用和所有依赖的运行环境。容器是轻量级、独立的，可以快速启动和停止。

镜像（Image）：是容器的静态模板，包含应用的所有依赖、环境变量、库文件等。容器是由镜像启动的，每次启动都会基于镜像的状态创建一个新的容器实例。

Docker 引擎（Docker Engine）：是一个客户端-服务器的应用，提供了容器化技术。它包括一个守护进程（Docker Daemon）、一个 REST API 和一个命令行界面（CLI）。Docker 引擎负责容器的创建、管理和运行。

数据卷（Volumes）：用于持久化和共享容器中的数据。容器退出后，数据卷中的数据仍然存在，可以跨容器使用。

Docker Hub：是一个公共的 Docker 镜像仓库，用户可以从中获取或上传镜像，类似的镜像仓库还有很多，不一一列举了。

Docker与传统虚拟化(VM)的区别？

虚拟化技术：传统的虚拟化技术通过在硬件上运行多个虚拟机，每个虚拟机都包括操作系统和应用程序。每个虚拟机都需要大量的资源，并且启动时间较长。

Docker 容器化技术：容器技术通过共享宿主机的操作系统内核来提供虚拟化，它是操作系统级别的虚拟化。每个容器都包含应用及其依赖，但不需要额外的操作系统。容器更轻量级，启动速度快。

Docker 的优势

从与传统虚拟化技术进行对比可以明显看出docker的优势所在：

  • 轻量级：容器使用宿主机操作系统的内核，因此比虚拟机更小，启动更快，资源消耗更少。
  • 可移植性：容器包含了应用及其所有依赖，可以在任何支持 Docker 的环境中运行，确保跨平台的一致性。
  • 快速部署和高效开发：由于容器化的隔离特性，开发人员可以在本地快速部署应用，进行测试和调试，然后直接将容器迁移到生产环境。
  • 扩展性：Docker 可以轻松地通过 Docker Compose、Kubernetes 等工具进行集群管理，支持大规模的容器部署。

基础技术

Linux Namespace

Linux Namespace是Linux Kernel的一个功能，它可以隔离一系列的系统资源，比如PID（Process ID）、User ID、Network等。一般看到这里，很多人会想到一个命令chroot，就像chroot允许把当前目录变成根目录一样（被隔离开来的），Namespace也可以在一些资源上，将进程隔离起来，这些资源包括进程树、网络接口、挂载点等。

简单介绍

Linux Namespace 是 Linux 内核提供的一种机制，用于实现资源的隔离。通过 Namespace，可以将系统的全局资源（如进程、网络、文件系统等）隔离成不同的命名空间，使得每个命名空间中的资源相互独立，互不干扰。Docker 正是利用 Namespace 技术来实现容器的资源隔离。

Linux 提供的 Namespace 类型

Namespace 类型 隔离的资源 作用
PID Namespace 进程 ID (PID) 不同 Namespace 中的进程可以有相同的 PID，实现进程 ID 的隔离。
Mount Namespace 文件系统挂载点 不同 Namespace 中的文件系统挂载点相互独立，实现文件系统的隔离。
UTS Namespace 主机名和域名 不同 Namespace 中的主机名和域名可以不同，实现主机名和域名的隔离。
IPC Namespace 进程间通信 (IPC) 资源，如消息队列、信号量、共享内存等 不同 Namespace 中的进程无法直接通过 IPC 通信，实现 IPC 资源的隔离。
Network Namespace 网络设备、IP 地址、端口等网络资源 不同 Namespace 中的网络栈相互独立，实现网络资源的隔离。
User Namespace 用户和用户组 ID (UID 和 GID) 不同 Namespace 中的用户和用户组可以有相同的 UID 和 GID，实现用户权限的隔离。

以PID Namespace为例

命名空间建立系统的不同视图，从用户的角度来看，每一个命名空间应该像一台单独的Linux计算机一样，有自己的init进程（PID为1），其他进程的PID依次递增，A和B空间都有PID为1的init进程，子命名空间的进程映射到父命名空间的进程上，父命名空间可以知道每一个子命名空间的运行状态，而子命名空间与子命名空间之间是隔离的。从图2.1所示的PID映射关系图中可以看到，进程3在父命名空间中的PID为3，但是在子命名空间内，它的PID就是1。也就是说用户从子命名空间A内看进程3就像init进程一样，以为这个进程是自己的初始化进程，但是从整个host来看，它其实只是3号进程虚拟化出来的一个空间而已。

Linux Cgroups

讲述的是构建Linux容器的Namespace技术，它帮助进程隔离出自己单独的空间，但Docker是怎么限制每个空间的大小，保证它们不会互相争抢的呢？这就要用到Linux的Cgroups技术

简单介绍

Cgroups（Control Groups） 是 Linux 内核提供的一种机制，用于限制、控制和监控进程组的资源使用。通过 Cgroups，可以对进程组的 CPU、内存、磁盘 I/O、网络等资源进行精细化的管理和分配。Cgroups 是容器技术（如 Docker）实现资源隔离和限制的核心技术之一。

Cgroups 的核心概念

子系统（Subsystem）

  • Cgroups 通过子系统来管理不同类型的资源。每个子系统负责管理一种资源。

  • 常见的子系统包括：

    • cpu：限制 CPU 使用。
    • memory：限制内存使用。
    • blkio：限制块设备 I/O。
    • net_cls：标记网络数据包，用于流量控制。
    • devices：控制设备访问权限。
    • freezer：挂起或恢复进程组。

    每个subsystem会关联到定义了相应限制的cgroup上，并对这个cgroup中的进程做相应的限制和控制。

控制组（Control Group）
  • 控制组是 Cgroups 的基本单位，每个控制组可以包含一组进程。
  • 每个控制组可以绑定一个或多个子系统，并为这些子系统设置资源限制。
层级结构（Hierarchy）
  • Cgroups 以树形结构组织控制组，形成一个层级结构。
  • 每个层级结构可以绑定一个或多个子系统。
  • 子控制组会继承父控制组的资源限制。

hierarchy的功能是把一组cgroup串成一个树状的结构，一个这样的树便是一个hierarchy，通过这种树状结构，Cgroups可以做到继承。比如，系统对一组定时的任务进程通过cgroup1限制了CPU的使用率，然后其中有一个定时dump日志的进程还需要限制磁盘IO，为了避免限制了磁盘IO之后影响到其他进程，就可以创建cgroup2，使其继承于cgroup1并限制磁盘的IO，这样cgroup2便继承了cgroup1中对CPU使用率的限制，并且增加了磁盘IO的限制而不影响到cgroup1中的其他进程。

三个组件相互的关系
  1. 层级结构绑定子系统：
    • 每个层级结构可以绑定一个或多个子系统。
    • 例如，一个层级结构可以绑定 cpu 和 memory 子系统，用于同时管理 CPU 和内存资源。
  2. 控制组属于层级结构：
    • 控制组是层级结构中的节点，每个控制组属于某个层级结构。
    • 控制组会继承层级结构绑定的子系统，并可以设置具体的资源限制。
  3. 进程属于控制组：
    • 进程被添加到某个控制组中，受到该控制组的资源限制。
    • 进程的资源使用情况会被子系统和层级结构监控和管理。

Union File System

Union File System，简称UnionFS，是一种为Linux、FreeBSD和NetBSD操作系统设计的，把其他文件系统联合到一个联合挂载点的文件系统服务。它使用branch把不同文件系统的文件和目录“透明地”覆盖，形成一个单一一致的文件系统。这些branch或者是read-only的，或者是read-write的，所以当对这个虚拟后的联合文件系统进行写操作的时候，系统是真正写到了一个新的文件中。看起来这个虚拟后的联合文件系统是可以对任何文件进行操作的，但是其实它并没有改变原来的文件，这是因为unionfs用到了一个重要的资源管理技术，叫写时复制。

写时复制（copy-on-write，下文简称CoW），也叫隐式共享，是一种对可修改资源实现高效复制的资源管理技术。它的思想是，如果一个资源是重复的，但没有任何修改，这时并不需要立即创建一个新的资源，这个资源可以被新旧实例共享。创建新资源发生在第一次写操作，也就是对资源进行修改的时候。通过这种资源共享的方式，可以显著地减少未修改资源复制带来的消耗，但是也会在进行资源修改时增加小部分的开销。

AUFS介绍

AUFS（Advanced Multi-Layered Unification Filesystem，高级多层统一文件系统）是一种联合文件系统，它允许你将多个目录（称为分支）挂载到同一个挂载点上，并以一种统一的方式呈现这些目录中的文件。AUFS 的主要特点是支持写时复制（Copy-on-Write, CoW），这意味着只有在文件被修改时才会复制文件，而不是在文件被读取时。

AUFS 的核心功能包括：

  1. 分层存储：多个目录（层）可以叠加在一起，形成一个统一的视图。
  2. 写时复制：当文件被修改时，AUFS 会复制该文件到可写层，而不是直接修改原始文件。
  3. 高效存储：多个容器可以共享相同的只读层，从而节省存储空间。

AUFS 是 Docker 早期版本中默认使用的存储驱动，但由于其未被合并到 Linux 内核主线，Docker 逐渐转向了其他存储驱动（如 overlay2）。

Docker 如何使用 AUFS

Docker 使用 AUFS 来实现镜像和容器的分层存储。

镜像分层：

  • Docker 镜像由多个只读层（Read-Only Layers）组成，每一层代表镜像的一个变更（例如安装软件包、添加文件等）。
  • 这些只读层通过 AUFS 叠加在一起，形成一个统一的文件系统视图。

容器层：

  • 当容器启动时，Docker 会在镜像的最上层添加一个可写层（Read-Write Layer）。
  • 容器的所有修改（如创建、修改或删除文件）都会记录在这个可写层中，而不会影响底层的只读镜像。

写时复制（CoW）：

  • 当容器需要修改一个文件时，AUFS 会将该文件从只读层复制到可写层，然后在可写层中进行修改。
  • 这种方式确保了镜像的不可变性，同时允许多个容器共享同一个镜像层。

高效存储：

  • 多个容器可以共享同一个镜像的只读层，从而减少存储空间的占用。
  • 只有容器的可写层是独立的，因此容器的启动速度非常快。

image layer和AUFS

每一个Docker image都是由一系列read-only layer组成的。image layer的内容都存储在Docker hosts filesystem的/var/lib/docker/aufs/diff目录下。而/var/lib/docker/aufs/layers目录，则存储着image layer如何堆栈这些layer的metadata。

container layer和AUFS

Docker使用AUFS的CoW技术来实现image layer共享和减少磁盘空间占用。CoW意味着一旦某个文件只有很小的部分有改动，AUFS也需要复制整个文件。这种设计会对容器性能产生一定的影响，尤其是在待复制的文件很大，或者位于很多image layer下方，又或者AUFS需要深度搜索目录结构树的时候。不过也不用过度担心，对于一个容器而言，每个image layer最多只需要复制一次。后续的改动都会在第一次拷贝的container layer上进行。

启动一个container的时候，Docker会为其创建一个read-only的init layer，用来存储与这个容器内环境相关的内容；Docker还会为其创建一个read-write的layer来执行所有写操作。container layer的mount目录也是/var/lib/docker/aufs/mnt。container的metadata和配置文件都存放在/var/lib/docker/containers/＜container-id＞目录中。container的read-write layer存储在/var/lib/docker/aufs/diff/目录下。即使容器停止，这个可读写层仍然存在，因而重启容器不会丢失数据，只有当一个容器被删除的时候，这个可读写层才会一起删除

镜像

镜像是一个只读的模板，它包含了应用程序的所有必要文件、库、环境配置等。可以看作是一个应用或服务的“蓝图”。它类似于一个操作系统的安装镜像，或者说是一辆车的设计图。

如果说容器是一个隔离的进程，它从哪里获取文件和配置？如何共享这些环境？

这就是容器镜像的用武之地。容器镜像是一个标准化的包，其中包含运行容器所需的所有文件、二进制文件、库和配置。

对于PostgreSQL 镜像，该镜像将打包数据库二进制文件、配置文件和其他依赖项。对于 Python Web 应用程序，它将包含 Python 运行时、您的应用程序代码及其所有依赖项。

镜像有两个重要原则

  1. 镜像是不可变的。创建镜像后，无法对其进行修改。您只能创建新镜像或在其上添加更改。
  2. 容器镜像由层组成。每一层都表示一组添加、删除或修改文件的文件系统更改。

什么是Dockerfile

Dockerfile 是一个文本文件，包含了一系列指令，用于定义如何构建一个 Docker 镜像。通过编写 Dockerfile，你可以描述从基础镜像到安装依赖、设置环境变量、复制文件、执行命令等步骤，最终生成一个自定义的 Docker 镜像。

常见 Dockerfile 指令

指令 作用
FROM 指定基础镜像，例如 FROM ubuntu 或 FROM node:14
RUN 执行命令（例如安装软件包），通常用于设置容器的环境
COPY 从主机复制文件或目录到镜像中的指定位置
ADD 类似于 COPY，但具有额外的功能，如解压文件或从 URL 下载文件
WORKDIR 设置工作目录，所有后续命令将在该目录下运行
CMD 设置容器启动时默认执行的命令（可以被 docker run 时指定的命令覆盖）
ENTRYPOINT 设置容器启动时的主命令，通常与 CMD 组合使用
EXPOSE 指定容器会使用的端口，供外部访问
ENV 设置环境变量
VOLUME 创建挂载点，用于持久化存储

数据卷

一旦容器退出，容器可写层的所有内容都会被删除。那么，如果用户需要持久化容器里的部分数据该怎么办呢？volume就是用来解决这个问题的。

数据卷是 Docker 提供的一种机制，用于持久化容器数据。数据卷存储在宿主机的文件系统中，容器通过挂载数据卷来访问数据。数据卷独立于容器的生命周期，它不受容器退出或删除的影响，即使容器被删除，数据卷中的数据依然保留。

Docker 数据卷（Volume）用于持久化容器的数据，它可以在容器之间共享，并且即使容器删除，数据也能保留。以下是与 Docker 数据卷相关的所有常用命令：

Docker 数据卷命令

命令 作用
docker volume ls 列出所有数据卷
docker volume inspect <VOLUME_NAME> 查看指定数据卷的详细信息
docker volume create <VOLUME_NAME> 创建一个新的数据卷
docker volume rm <VOLUME_NAME> 删除指定的数据卷
docker volume prune 删除所有未使用的挂载卷（注意：会删除未被容器使用的卷）
docker volume ls --filter <FILTER> 使用过滤器列出数据卷，例如通过标签或名称过滤
docker run -v <VOLUME_NAME>:<CONTAINER_PATH> 在容器中挂载指定的数据卷
docker run --mount source=<VOLUME_NAME>,target=<CONTAINER_PATH> 使用 --mount 语法挂载卷，推荐用于现代 Docker 版本
docker volume mount <VOLUME_NAME> 挂载指定的数据卷（一般是通过 docker run 命令实现挂载）

  • 创建卷： 数据卷可以在容器创建时自动创建，或者使用 docker volume create 显式创建。

容器

容器是镜像的运行时实例。它是在镜像的基础上启动出来的，带有自己的文件系统、进程、网络等资源。容器可以看作是一个活跃的、运行中的应用。容器就像是从蓝图（镜像）上建造出来的一辆车，它会具体化为一个可以行驶的实体。

容器（Container）是一个轻量级、可移植、可执行的软件包，它封装了应用及其所有依赖项（包括操作系统库和配置文件），能够在任何环境中运行。容器的主要特点是隔离性和高效性。容器与虚拟机的区别在于它们共享宿主机的操作系统内核，而虚拟机是通过虚拟化实现完整的操作系统模拟。

容器的核心特点：

  • 隔离性：每个容器都有自己的文件系统、进程空间、网络配置等，但都在同一个宿主操作系统内核上运行。
  • 高效性：容器不需要模拟整个操作系统，资源消耗较低，启动速度较快。
  • 可移植性：容器镜像可以在任何支持Docker的环境中运行，保证了一致性。
  • 易于管理：容器的创建、启动、停止、销毁等操作都非常简便，适合微服务架构的应用部署。
