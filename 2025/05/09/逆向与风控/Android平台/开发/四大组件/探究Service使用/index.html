探究Service使用

Service与多线程联系

在之前的线程与多线程机制 - The Peak Tower源码阅读中简单探讨过在Android系统下多线程开发机制，这里简单的回顾一下：

Android 多线程编程是为了在主线程（UI 线程）之外执行耗时操作，以避免阻塞 UI 更新和用户交互，从而提升应用的响应性和性能。Android 提供了多种多线程机制，其中最常用的是 HandlerThread、AsyncTask（已弃用）、Executors 和 Coroutines（Kotlin）。HandlerThread 是一个带有 Looper 的线程，可以与 Handler 结合使用，实现线程间的通信；AsyncTask 曾用于在后台执行任务并更新 UI，但由于其容易导致内存泄漏和复杂性，现已被弃用；Executors 提供了线程池管理，可以高效地执行多个任务；Coroutines 是 Kotlin 提供的轻量级并发解决方案，通过挂起和恢复机制简化了异步编程。多线程的核心原理是通过将耗时操作（如网络请求、数据库查询）放到后台线程执行，同时通过回调或消息机制将结果传递回主线程更新 UI，从而避免主线程被阻塞。开发者需注意线程安全问题，避免多个线程同时访问共享资源导致数据不一致，常用的同步机制包括 synchronized、ReentrantLock 和原子类等。通过合理使用多线程，可以显著提升应用的性能和用户体验。

这与本文提到了Service有什么联系呢？这就要回归到对Service使用的定义上来了：

多线程机制（如 Thread、HandlerThread、Executors、Coroutines）主要用于在应用程序内部实现并发执行，避免阻塞主线程，适合处理短期的、与 UI 无关的任务（如网络请求、文件读写），其生命周期通常与 Activity 或 Fragment 绑定；

Service 是 Android 四大组件之一，专门用于在后台执行长时间运行的任务（如音乐播放、文件下载），即使应用退出或用户切换到其他应用，Service 仍可继续运行。Service 内部通常会结合多线程机制来实现具体的任务执行，例如在 Service 中创建线程或使用 IntentService（已弃用，推荐使用 WorkManager）来处理耗时操作。因此，多线程机制是 Service 实现后台任务的技术基础，而 Service 则提供了更高级的任务管理和生命周期控制，使后台任务能够独立于 UI 组件运行。

Activity 与 Service 通信

通过 bindService() 绑定 Service，并使用 Binder 实现通信。

Activity与Service之间的交流媒介是Service中的onBind()方法，它带有Intent参数，并且返回值为 IBinder，也就是说，可以通过IBinder来传递数据。

实现步骤如下：

  1. 在自定义Service类中，自定义一个Binder类，将要暴露的方法写到该类中，同时实例化这个类。
  2. 然后重写onBind()方法，将这个Binder对象返回。
  3. 在调用该Service的Activity中，实例化一个ServiceConnection对象，重写onServiceConnected()方法来获取Binder对象，之后调用相关方法即可。

两种启动方式的区别

StartService

系统对同一个Service只会创建一个Service实例。如果Service实例不存在，调用startService()方法则会实例化一个Service对象，否则复用之前创建的Service实例。通过StopService()关闭Service。

如果调用者直接退出而没有调用StopService()方法，Service会一直在后台运行。该Service的调用者再启动起来后可以通过StopService()关闭Service。

BindService

系统对同一个Service只会创建一个Service实例。如果Service实例不存在，调用bindService则会实例化一个Service对象，否则复用之前创建的Service实例。

当Service只与一个客户端绑定时，调用unbindService()或者调用者退出，Service会被销毁。当Service与多个客户端绑定时，只有与所有客户端取消绑定后，Service才会被销毁。

**startService()用于启动独立的后台任务，适合不需要与启动组件通信的场景。bindService()**用于与 Service 建立双向通信，适合需要频繁交互的场景。
