So加载和链接(linker)

本文以android 9源码进行分析

命名空间(namespace机制)

主要参考Android 8.0 - 基于命名空间的动态链接—— 隔离 Android 中应用程序和系统的本地库 - 张同光 - 博客园总结来说：

概念

命名空间是为了在 Android 操作系统中管理动态链接库（so 库）的加载和共享，确保系统和应用程序的本地库（so 库）得到有效的隔离和管理。这项机制的核心目标是提高系统的安全性、稳定性，并防止应用程序滥用系统资源，特别是访问或修改不应该访问的库。

概念 描述 作用 相关命名空间
隔离 每个命名空间中的库被隔离，防止它们之间的冲突和无意间的相互干扰。 确保每个命名空间只加载所需的库，应用程序库与系统库相互隔离，避免不必要的干扰。 系统命名空间、应用程序命名空间、VNDK 命名空间、sphal 命名空间等
共享 通过命名空间链接，不同命名空间之间可以共享某些库，但共享是经过控制的。 仅允许特定的库（如公共系统库）在多个命名空间之间共享，减少重复加载的开销，确保安全性和稳定性。 系统命名空间与应用程序命名空间、VNDK 命名空间、sphal 命名空间等
灰名单（Greylist） 允许应用访问系统的私有库，但这种访问是受限制的，且带有性能和安全风险。 应用可以访问一些被标记为不安全或过时的库，但这种访问受到限制，且会增加内存开销和潜在安全风险。 系统命名空间、应用程序命名空间
VNDK 命名空间 用于管理供应商提供的本地库，隔离供应商层的库与系统框架库。 确保供应商库与系统库隔离，减少不同版本的供应商库与系统框架库之间的冲突，提升系统的稳定性和兼容性。 VNDK 命名空间
sphal 命名空间 用于管理硬件抽象层（HAL）相关的库，提供系统与供应商的硬件接口。 管理硬件抽象层库，允许系统与供应商之间的硬件接口正确交互，同时与其他命名空间隔离，避免冲突。 sphal 命名空间

应用要加载 SO 库的流程

步骤 描述 关键函数
1. 查找共享库文件 系统系统根据指定的路径查找库文件，包括应用的私有库路径和系统库路径。 load_library()
2. 映射库到内存 使用 mmap 或类似技术将库映射到内存中。 load_library()
3. 创建 soinfo 结构体 为每个加载的库创建并初始化 soinfo 结构体，存储库的基本信息、动态段、符号表等。 soinfo
4. 解析动态段 读取 ELF 文件的动态段（dynamic），解析动态符号、重定位信息等。 link_image()
5. 符号解析 查找并解析库中的符号（函数、变量等），处理符号依赖关系。 dlsym()
6. 重定位 根据重定位表（rel）更新符号地址。 relocate()
7. 执行库初始化 执行库中的初始化代码，例如 _init 或构造函数。 call_constructors()

命名空间类型

命名空间类别 描述 用途 示例
系统命名空间 默认命名空间，包含所有系统共享的库。 加载操作系统和框架的共享库，供系统进程和应用程序使用。 libc.so、libm.so、libandroid.so 等
应用程序命名空间 每个 Android 应用程序的私有命名空间。 加载应用程序自己的共享库，确保库的隔离和应用私有性。 应用程序的私有 .so 库
VNDK 命名空间 用于管理供应商库（Vendor Native Development Kit）。 隔离供应商代码和框架代码，确保系统稳定性，兼容性。 libhidl.so、libstagefright.so 等供应商库
SPHAL 命名空间 系统与供应商之间的硬件抽象层库。 管理硬件抽象层（HAL）库，允许系统与供应商硬件库正确交互。 libhwbinder.so 等 HAL 相关库
Greylist 命名空间 包含灰名单库，允许有限制地访问某些私有或过时的库。 允许应用访问某些系统库（灰名单库），但访问受限制，通常用于过时或即将弃用的功能。 被标记为灰名单的系统库（如特定版本的 libc）
供应商命名空间 用于管理设备供应商特定的库。 供应商层的库，仅在供应商定制化的代码中使用。 硬件解码库、传感器库等硬件特定的库

JAVA层调用SO

Android 应用程序通过 System.loadLibrary() 或 Runtime.loadLibrary() 加载本地库。这些本地库通常是 .so 文件，它们包含了通过 JNI 与 Java 代码交互的本地方法。Android 的类加载机制和虚拟机（ART）会负责解析这些库，并将其加载到内存中。

LoadLibrary()

http://androidxref.com/9.0.0_r3/xref/libcore/ojluni/src/main/java/java/lang/System.java#1651

在 Java 中，Runtime 是一个代表当前 Java 应用程序运行时环境的类。它提供了与 Java 虚拟机（JVM）进行交互的多种功能，如内存管理、运行时参数管理、执行系统命令等。

**loadLibrary0**：

  • 这是 Runtime 类的一个本地方法，实际上是通过 JNI (Java Native Interface) 调用底层操作系统的功能来加载共享库。在 Runtime.getRuntime().loadLibrary0() 中，loadLibrary0 负责执行具体的库加载工作，通常由 JVM 实现和操作系统平台提供支持。

**VMStack.getCallingClassLoader()**：

  • 这个方法返回调用当前方法的类加载器（ClassLoader）。类加载器负责加载类和资源，而 VMStack.getCallingClassLoader() 是获取调用当前方法的类加载器。该加载器通常是调用 loadLibrary 方法的类的 ClassLoader。在 Android 中，VMStack.getCallingClassLoader() 通过获取调用当前方法的类加载器来确定在当前上下文中加载共享库的路径。Android 中的类加载器负责加载 Java 类，并且在加载库时，会根据不同的 ClassLoader 来决定如何查找本地库（包括从 lib 目录、apk 文件中等）。

步骤 描述
1. 检查库是否已加载 函数首先检查是否已经加载该库。如果库已加载，并且类加载器相同，则返回成功；如果类加载器不同，生成错误信息并返回 false。
2. 解码类加载器 解码传入的类加载器对象，并检查是否为 BootClassLoader。如果是，设置为 nullptr，否则获取类加载器的分配器。
3. 打开共享库 调用 android::OpenNativeLibrary 使用 dlopen 加载共享库。如果加载失败，返回 false，并在 error_msg 中记录错误。
4. 创建共享库对象 创建 SharedLibrary 对象表示已加载的库，并通过加锁保证线程安全，将库添加到全局库列表中。
5. 调用 JNI_OnLoad 检查库中是否包含 JNI_OnLoad 函数，如果有，调用它进行初始化。如果返回错误版本，记录错误信息；否则标记为成功。
6. 返回加载结果 设置共享库的加载结果并返回是否成功。

**__builtin_return_address(0)**：

  • 这是一个 GCC 提供的内建函数，用来获取当前函数的调用者的返回地址。
  • 传入的参数 0 表示获取当前函数的调用者的返回地址。如果传入 1，则表示获取当前调用者的调用者的返回地址。
  • 该返回地址可以用来了解调用栈的位置，通常用于调试或记录调用路径。

**__loader_dlopen**：

  • 这是 dlopen 的底层实现，负责加载共享库。通过传递 caller_addr，它可能会记录加载请求的调用源，或者用于进一步的调试/分析。
  • __loader_dlopen 可能是一个与操作系统或具体实现相关的自定义函数，通常用于封装 dlopen 的实际底层操作。

**dlopen**：

  • dlopen 是动态库加载的标准函数，用于加载共享库（.so 文件）到当前进程的内存空间中。
  • filename 是共享库的路径，flag 用于指定加载的方式（例如：是否立即绑定符号，是否允许卸载等）。

dlsym()

http://androidxref.com/9.0.0_r3/xref/art/runtime/ti/agent.cc#154

在 Android 和 Linux 系统中，dlsym() 是一个用于动态获取共享库中符号地址的函数，它是 POSIX 标准的一部分，主要用于在运行时动态加载共享库并解析其中的符号（函数或变量）。并返回该符号的地址。这通常用于在程序运行时动态加载库，并通过符号名称调用库中的函数或访问库中的变量。

dlsym() 查找符号时，并不直接支持 C++ 命名空间。也就是说，如果一个符号在 C++ 中被定义在某个命名空间中，dlsym() 需要使用符号的完全限定名称来查找。比如：mynamespace::my_function 来查找符号。这通常是因为 C++ 编译器会将符号的名称进行修饰（name mangling）以区分同名的符号

先了解一下LoadTask 类

LoadTask 类管理了一个库的加载过程，其中包括：

  1. **create()**：通过工厂方法创建 LoadTask 实例。
  2. **read()**：读取 ELF 文件的内容。
  3. **load()**：加载 ELF 文件到内存，并填充相关的库信息（如基地址、大小等）。
  4. 通过 ElfReader 类来处理 ELF 格式文件的加载和解析。

下文的task就是LoadTask 类的实例。

加载库是动态加载、解决依赖关系的过程，通常在运行时进行；而预加载库是在程序启动时提前加载，确保一些库在程序启动时就已经准备好，避免后续的延迟和重复加载。
