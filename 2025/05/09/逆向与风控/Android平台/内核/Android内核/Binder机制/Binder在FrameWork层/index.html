其中 IBinder是framework层为应用程序开发提供的定义跨进程通信的基本接口。其包括实现的实现类有：

  • Binder：服务端的实现类，用于处理客户端的请求。
  • BinderProxy：客户端的代理类，用于向服务端发送请求。

在前面服务的注册与获取 - The Peak Tower等文章中，我们着重谈到了在Native层的进程间通信机制。本文将介绍Binder在Android Framework层通过JNI被调用的机制。

在JNI机制分析 - The Peak Tower这篇文章中我们谈到：

在android系统启动流程 - The Peak Tower中谈到了Android系统在启动启动过程中，先启动Kernel创建init进程，紧接着由init进程fork第一个横穿Java和C/C++的进程，即Zygote进程。Zygote启动过程中会AndroidRuntime.cpp中的startVm创建虚拟机，VM创建完成后，紧接着调用startReg完成虚拟机中的JNI方法注册。接着在register_jni_procs(gRegJNI, NELEM(gRegJNI), env)方法中循环调用gRegJNI数组成员所对应的方法。而gRegJNI数组，有100多个成员变量，每个成员都代表一个类文件的jni映射

startReg

Cross Reference: /frameworks/base/core/jni/AndroidRuntime.cpp

其中gRegJNI是一个数组，记录所有需要注册的jni方法，其中有一项便是REG_JNI(register_android_os_Binder)

register_android_os_Binder

通过 JNI 将多个与 Binder 机制相关的 Java 类的字段和方法与本地代码关联起来。它通过查找类、获取方法和字段 ID，并注册本地方法，实现了 Java 和本地代码之间的交互。

android.os.Binder：注册 execTransact 方法和 mObject 字段。

android.os.BinderInternal：注册底层功能方法，如线程池管理。

android.os.BinderProxy：注册事务处理和远程对象引用管理方法。

int_register_android_os_Binder

Cross Reference: /frameworks/base/core/jni/android_util_Binder.cpp

int_register_android_os_Binder方法的主要功能：

  • 通过gBinderOffsets，保存Java层Binder类的信息，为JNI层访问Java层提供通道；
  • 通过RegisterMethodsOrDie，将gBinderMethods数组完成映射关系，从而为Java层访问JNI层提供通道。

也就是说该过程建立了Binder类在Native层与framework层之间的相互调用的桥梁。

int_register_android_os_BinderInternal

建立了是BinderInternal类在Native层与framework层之间的相互调用的桥梁。

int_register_android_os_BinderProxy

建立了是BinderProxy类在Native层与framework层之间的相互调用的桥梁。
