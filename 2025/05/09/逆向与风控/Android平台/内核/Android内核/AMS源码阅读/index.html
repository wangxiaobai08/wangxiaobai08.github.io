AMS源码浅析

在android系统启动流程 - The Peak Tower中，我们简单谈到：

Android系统在启动的时候,在启动两个重要的进程，一个是Zygote进程，另一个是由zygote进程fork出来的system_server进程。SystemSever负责启动系统的各项服务。其中各项服务中包括：

Activity Manager Service (AMS) 管理应用程序的生命周期，包括启动和停止应用、管理任务和活动栈、处理广播等
AMS（Activity Manager Service） 是 Activity 管理服务 的核心组件，负责管理应用程序的 生命周期、任务栈（Task）、进程调度 以及 四大组件（Activity、Service、BroadcastReceiver、ContentProvider） 的交互。

这篇文章主要从四大组件的角度，简单高屋建瓴的聊聊AMS主要干了什么事？

开始之前先简单区分一下ActivityThread和ApplicationThread的区别

ActivityThread

  • 主线程管理者：代表应用进程的主线程（UI 线程），是应用进程的入口类（main() 方法所在）。
  • 生命周期调度：直接处理 AMS 的指令，驱动 Activity/Service 的生命周期（如 onCreate(), onResume()）。
  • 全局上下文：持有 Application 对象和资源管理相关组件（如 Instrumentation）。

ApplicationThread

  • Binder 通信桥梁：是 ActivityThread 的内部类，继承 IApplicationThread.Stub，作为 Binder 服务端 供 AMS 回调。
  • 被动响应：接收 AMS 的跨进程请求（如启动 Activity），转交给 ActivityThread 执行，本身不处理逻辑。

AMS 与 APP 进程之间通信

在APP安装到启动流程大体概览 - The Peak Tower中我们对于在桌面上APP点击流程有了大概的了解：

当 Zygote进程 fork()出一个 app(应用程序进程) 进程后，就没 Launcher 和 Zygote 什么事了。接下来就是 AMS 与 APP(应用程序进程) 进程之间打交道了。

注意：在 Android 中，AMS（Activity Manager Service） 与 App 进程 属于不同的进程，它们通过 Binder IPC 机制 实现双向通信

App → AMS 的通信
AMS 在系统启动时通过 ServiceManager.addService("activity", ams) 注册到 ServiceManager（SM）。App 通过 ServiceManager.getService("activity") 获取 AMS 的 Binder 代理对象（IActivityManager），从而调用 AMS 的接口（如 startActivity）。

AMS → App 的通信
App 进程启动后，会向 AMS 发送自己的 Binder 代理对象（IApplicationThread，即 ApplicationThreadProxy）。AMS 保存该代理，后续通过它主动控制 App 的生命周期（如触发 onCreate、onResume）。

其中核心的结构体如下:

在这张图中，绿色的部分是在SDK中开放给应用程序开发人员的接口，蓝色的部分是一个典型的Proxy模式，红色的部分是底层的服务实现，是真正的动作执行者。这里的一个核心思想是Proxy模式：

通过 ActivityManagerProxy（代理类）隐藏 ActivityManagerService（真实服务）的实现细节，客户端（如 ActivityManager）只需与代理交互，无需直接依赖底层服务，降低耦合性。
代理类与真实服务均实现 IActivityManager 接口，调用者无法感知操作的是本地代理还是远程服务，保证接口一致性。

  • 代理层（Proxy）：
    ActivityManagerProxy 作为中间层，将客户端的请求通过 Binder 转发给远端的 ActivityManagerService。
  • 服务端（Service）：
    ActivityManagerService 是真正的执行者，运行在 system_server 进程，处理跨进程请求（如启动 Activity）。

管理Activity

Activity 管理的核心组件

组件 存储结构 作用 关键关联
ActivityRecord 单个 Activity 的抽象 存储 Activity 的所有信息（如 Intent、状态、进程、窗口Token）。 与 Activity 一一对应，由 ActivityStarter 创建。
TaskRecord ArrayList<ActivityRecord> 表示一个任务栈（Task），按 先进后出 管理多个 Activity（如应用内的返回栈）。 归属某个 ActivityStack。
ActivityStack ArrayList<TaskRecord> 管理多个 Task（如 主栈、全栈栈），处理栈内 Activity 的调度（如 RESUME、PAUSE）。 由 ActivityStackSupervisor 管理。
ActivityStackSupervisor 管理所有 ActivityStack 协调多个栈的交互（如分屏模式、多窗口），决策哪个栈/Activity 应获得焦点。 直接受 AMS 控制。

管理Service

详细参考：

startService启动过程分析 - Gityuan博客 | 袁辉辉的技术博客

四大组件之ServiceRecord - Gityuan博客 | 袁辉辉的技术博客

Android四大组件之bindService源码实现详解_bindservice的实现-CSDN博客

每个app进程运行的Service, 对应于system_server进程中AMS的ServiceRecord对象. 也就是说所有app进程的Service, 都会记录在system_server进程。

Service 启动流程

Client 端：调用 startService() 或 bindService() → 通过 IActivityManager 通知 AMS。

AMS 端：

  • 创建 ServiceRecord → 关联到 ProcessRecord → 由 ActiveServices 管理生命周期。
  • 通过 ApplicationThread 回调客户端执行 onStartCommand() 或 onBind()。

其中Activity 受 任务栈（Task） 影响（如 launchMode），而 Service 仅依赖 进程（Process）。Service 没有“栈”概念，但绑定关系（bindService）会记录在 ServiceRecord 中。

Service 生命周期

同Activity 对比可以看出：

  • Activity 生命周期受 用户交互（如返回键、多任务切换）驱动，由 ActivityStack 管理。
  • Service 生命周期受 显式调用（如 startService()）或 绑定关系 驱动，由 ActiveServices 管理。

管理BroadcastReceiver

主要参考：四大组件之BroadcastRecord - Gityuan博客 | 袁辉辉的技术博客

Cross Reference: /frameworks/base/services/core/java/com/android/server/am/ActivityManagerService.java

类/数据结构 作用
ActivityManagerService 广播的注册、发送、分发的总入口。
BroadcastQueue 管理广播队列（前台/后台），处理超时和ANR。
BroadcastRecord 封装单次广播的所有信息（Intent、接收者列表等）。
ReceiverList 维护一个进程注册的所有动态广播接收者。
BroadcastFilter 关联 IntentFilter 和接收者（动态注册）。

注册 BroadcastReceiver

当应用通过 Context.registerReceiver() 或静态注册（AndroidManifest.xml）时，最终会调用到 AMS 的 registerReceiver 方法。

发送广播

通过 Context.sendBroadcast() 发送广播时，最终调用 AMS 的 broadcastIntent 方法。

  1. 权限检查：验证发送者是否有权限发送该广播。
  2. Intent匹配：通过 mReceiverResolver 查找匹配的 BroadcastFilter。
  3. 调度广播：将广播加入队列，按优先级和顺序分发。

广播分发与处理

Cross Reference: /frameworks/base/services/core/java/com/android/server/am/BroadcastQueue.java

AMS 通过 BroadcastQueue 管理广播的分发，核心逻辑在 BroadcastQueue 类中。

注意：

  1. 并行广播（无序广播）：
    • 直接并发分发到所有匹配的接收者（通过 processNextBroadcast 处理）。
  2. 串行广播（有序广播）：
    • 按优先级顺序逐个分发，前一个接收者的 onReceive() 执行完成后才会触发下一个。

管理ContentProvider

主要参考：四大组件之ContentProviderRecord - Gityuan博客 | 袁辉辉的技术博客

类/数据结构 作用
ContentProviderRecord 记录 ContentProvider 的元数据（Authority、进程、权限等）。
ProviderMap AMS 内部维护的全局 Provider 映射表（按类名和 Authority 索引）。
ProcessRecord 关联进程与它发布的 ContentProvider（通过 pubProviders 字段）。
ContentProviderHolder 跨进程传递的 ContentProvider 信息（包含 Binder 接口）。

安装 ContentProvider

启动时注册：

  • 应用进程通过 ActivityThread 调用 installProvider 安装 ContentProvider。
  • AMS 通过 publishContentProviders 接收提供者列表并更新全局记录。

AMS 记录 Provider：

  • 每个 ContentProvider 对应一个 ContentProviderRecord，存储在 AMS 的 mProviderMap 中。

客户端访问 ContentProvider

解析 Authority：

  • 客户端调用 ContentResolver.query()，通过 acquireProvider 请求 AMS 获取 ContentProvider。

AMS 匹配 Provider：

  • AMS 从 mProviderMap 中查找匹配的 ContentProviderRecord。
  • 若目标 Provider 所在进程未启动，AMS 会先启动进程（如 startProcessLocked）

Provider 进程启动与绑定

若 ContentProvider 所在进程未运行，AMS 会启动进程并等待其发布 Provider。

  1. 启动进程：
    • AMS 调用 startProcessLocked 创建新进程，通过 zygote fork 应用进程。
  2. 进程初始化：
    • 进程启动后，ActivityThread.main() 被调用，执行 attachApplication 通知 AMS。
  3. 安装 Provider：
    • AMS 通过 attachApplication 将待安装的 Provider 列表发送到目标进程。
