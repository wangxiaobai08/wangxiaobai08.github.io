COM基础知识

COM接口

COM接口通常是一组函数的逻辑集合，COM对象可以提供多个COM接口，每个接口提供不同的服务，因此COM接口与COM对象一样，都是用GUID来标识的，客户通过GUID来获取接口指针，再通过接口指针获取对应的服务；标识COM接口的GUID称为：IID。COM接口的本质:

COM接口是纯虚类（抽象类），其核心特征：

  • 命名规范：以大写 I 开头（如 IMathCalculator）
  • 强制继承：必须直接或间接继承 IUnknown
  • 二进制标准：本质是函数指针表（vtable），与语言无关

接口与实现的关系:

概念 角色 示例
接口 定义行为契约（纯虚类） IMathCalculator
实现类 具体实现接口的C++类 class MathCalculator : public IMathCalculator
接口指针 客户端通过它调用方法（实际指向vtable） IMathCalculator* pCalc

COM对象

COM对象其实就类似于C++中的对象，也就是说某个类的实例，包含了一组数据和操作。在COM模型中，COM对象的位置对于客户来说是透明的，即客户代码不需要直接初始化一个COM对象，而是COM库通过一个全局标识码GUID去对其进行初始化工作。GUID是一个128位的标识符，基本保证了COM对象的唯一性，另外COM接口也是用GUID来标识的。

GUID

GUID，又称之为全局唯一标识符，有16个字节，共128位二进制数，可以保证全球范围内不会重复，标识COM对象的GUID称为：CLSID。

COM组件中，GUID的结构定义如下：

典型的GUID定义:

  • CLSID：标识COM类（如 CLSID_WordApplication）
  • IID：标识接口（如 IID_IUnknown）
  • 生成工具：guidgen.exe 或在线GUID生成器

COM对象的核心特性

特性 说明 类比C++
封装性 包含数据和方法，但外部只能通过接口访问 类似类的实例，但所有成员私有化
位置透明 客户端无需知道对象在本地（DLL）、跨进程（EXE）还是远程机器（DCOM） 类似指针，但实际可能是网络代理
GUID标识 每个COM对象/接口用128位全局唯一标识符（GUID）注册 类似类名，但绝对唯一

COM应用模型

客户/服务模型

COM的核心是客户端-服务器架构，其交互方式如下：

角色 职责 典型示例
客户端 通过接口调用服务，无需知道实现细节 Excel VBA调用Word组件
服务器 实现COM接口并提供功能，分为进程内(DLL)/本地进程外(EXE)/远程(DCOM)三种 数学计算组件、打印机驱动

位置透明性：客户端代码相同，无论服务在何处

接口契约：双方严格通过接口交互，避免直接依赖

客户通过COM创建函数获取到COM库，COM库再通过DllGetClassObject获取组件，再通过组件获取类工厂对象接口指针，并用这个指针来调用各个方法。

COM组件

实现类工厂对象

COM对象的创建是通过类工厂来完成的，类工厂是COM对象的生产基地，对应每一个COM类，都有一个类工厂专门用于该COM类的对象创建操作。

类厂本身也是一个COM对象，它支持一个特殊的接口IClassFactory，这个接口的定义如下：

我们可以看见它继承了IUnknown接口，并且多了两个成员函数：CreateInstance、LockServer。

  • CreateInstance函数是IClassFactory接口中最重要的函数，它用于创建相应的COM对象，因为每个类厂只针对特定的COM对象，所以CreateInstance成员函数知道该创建什么样的COM对象。
  • LockServer用于控制组件的生存周期，用于在多客户调用COM时，锁定COM，以免一个客户退出时销毁了COM，那么其他客户的调用将发生错误。

实现自动卸载

只有当组件满足以下两个条件才能被卸载：

  1. 组件中对象数为0；
  2. 类厂的锁计数为0。

COM中的CoFreeAllLibraries函数可以检测当前进程中的所有组件程序，当发现某个组件满足上面两个条件时，就调用FreeLibrary函数把该组件从内存中释放，有两个问题：

  1. 谁来调用CoFreeAllLibraries函数？是由客户来调用，一般在程序空闲的时候调用。
  2. CoFreeAllLibraries函数怎么知道满足了上面说的可以卸载的条件？需要组件导出一个DllCanUnloadNow函数，如果DllCanUnloadNow函数返回S_OK表示对象可以被卸载。

COM客户

COM客户从只知道CLSID到获取到接口指针必须经过两步：

  1. 得到该CLSID的类厂对象；
  2. 由类厂对象创建COM对象，返回接口指针给客户。

客户得到对象接口指针后，可以用指针调用接口的成员函数，还可以获得其他接口的指针，从而得到COM对象的所有服务。

ATL

ATL（Active Template Library）是微软开发的一个C++模板库，主要用于创建高效、轻量级的COM（Component Object Model）组件，它通过模板技术提供了创建COM对象所需的基础代码框架，显著减少了开发者需要编写的样板代码量，同时保持了高性能和灵活性。

ATL最初设计用于开发ActiveX控件，但如今广泛应用于各种COM组件的开发，包括Windows服务、DLL插件和自动化服务器等；ATL的核心优势在于其模板化的实现方式，它通过编译时多态而非运行时多态来提高效率，避免了传统C++多重继承带来的复杂性和性能开销；

ATL提供了诸如智能指针（CComPtr）、COM对象基类（CComObject）、接口映射宏（COM_INTERFACE_ENTRY）等工具，简化了COM接口的实现和管理；此外，ATL还内置了对双重接口（dual interface）、连接点（connection points）和属性页（property pages）等高级COM特性的支持；ATL的代码生成机制通过向导和宏大幅降低了开发难度，开发者只需专注于业务逻辑实现，而ATL负责处理COM对象的生命周期管理、引用计数、接口查询等底层细节；

在Windows开发生态中，ATL与MFC（Microsoft Foundation Classes）形成互补关系，MFC更适合快速开发大型应用程序框架，而ATL则更适合需要高性能和精细控制的COM组件开发；

现代Windows开发中，尽管.NET技术逐渐普及，但ATL仍然在某些高性能场景和系统级开发中占据重要地位，特别是对于需要直接与Windows系统深度集成的组件开发。
