访问频率反爬虫

案例练习网站：

Scrape | Book

Scrape | Movie

访问频率指的是单位时间内客户端向服务器端发出网络请求的次数，它是描述网络请求频繁程度的量。正常用户浏览网页的频率不会像爬虫程序那么高，开发者可以将访问频率过高的客户端视为爬虫程序。

降速爬取

由于爬虫请求次数过高从而触发了反爬虫，很自然的想到利用time.sleep()方法模拟请求间隔。在降速的前提下某种程度上【利用自动化测试框架天生访问速度慢的缺陷】来说使用selenium等自动化测试框架也是一种天然的方法。

分布式爬虫

实际上，爬虫总是希望请求频率越高越好，这样才能够在最短的时间内完成爬取任务。刚才使用的time.sleep(1)这种降低请求频率的方法并不是爬虫工程师最好的选择。面对根据IP地址实现的访问频率限制反爬虫，我们可以使用多台机器共同爬取。假如数据总量为5万条，目标网站限速为1 r/s，使用time.sleep(1)这种方式完成爬取任务需要耗费的时间约为13.9小时。此时将爬取机器从1台增加到10台（10个IP），那么爬取时间就会降低到1.39小时。这种使用多台机器共同爬取的方法称为多机爬取，如果这些机器分布在不同的地域，并且它们使用的是相同的URL队列，那么这种爬虫组合就是分布式爬虫。分布式爬虫分为对等分布式和主从分布式

谈到分布式式爬虫就不得不提到scrapy-redis了。

scrapy-redis

【GitHub - rmax/scrapy-redis: Redis-based components for Scrapy.】

在前面的文章Scrapy初步使用 - The Peak Tower中简单谈到了scrapy的简单使用和原理，Redis在前面的专栏数据库中也分析过。而Scrapy-Redis是一个将 Scrapy 和 Redis 集成的库，旨在帮助开发者使用 Scrapy 在分布式爬虫架构下运行爬虫。它通过将 Scrapy 的数据存储、请求队列和去重功能与 Redis 集成，使得多个爬虫实例可以共享队列并协同工作，进而构建一个分布式爬虫系统。注意：scrapy-redis只是替换了redis的几个组件，不是一个新的框架。

这里主要围绕着两者如何结合的原理和基本使用。

scrapy-redis 的主要功能：

功能 说明
分布式请求队列 使用 Redis 存储请求队列，多个爬虫实例从 Redis 中获取请求进行爬取，实现并行处理，提高爬取效率。
分布式去重 基于 Redis 存储已抓取的 URL，避免重复抓取同一 URL，克服传统 Scrapy 去重的局限性。
分布式存储 爬虫抓取的结果存储到 Redis 中，支持数据流式传输，便于其他应用程序使用。
自动化任务分配 通过 Redis 实现爬虫任务的自动化分配，使得多台机器上的爬虫实例能够协同工作，分担任务。

整体架构

首先先回顾一个scrapy的架构：

scrapy会将生成的request放在队列中，这个队列是有deque实现的本地爬取队列queue。放入队列后，调度器scheduler才会调度里面的request，将request交给downloader执行。简单的调度架构如下图：

而分布式爬虫的简单架构如图：

从图中可以看出调度器scheduler可以扩展多个，因此download也可以扩展多个，而爬取队列queue始终为一个。这就是共享爬取队列。这就可以做到一个scheduler从队列中调度某个request时，其他的scheduler不会重复调度这个request，就可以做到多个scheduler同时爬取，这就是分布式爬虫的基本架构。

理解了这个大概原理就可以进行下一步scrapy_redis和分布式的结合：
scrapy_redis是scrapy框架的一个组件，因为scrapy并不具备分布式的实现，因此scrapy_redis重写了一些scrapy框架里的类，让他能够支持分布式爬虫的创建。

scrapy_redis和scrapy示意图：

scrapy_redis的核心就是redis数据库，因为scrapy_redis改写了scrapy中的queue队列，用redis数据库代替了queue，让本不能共享的queue变成了可以共享的数据库redis。因此，scrapy_redis将每一个request存入redis数据库中，这样就可以多个spider同时访问一个redis，进行分布式爬取。

scrapy_redis的流程

  • 在scrapy_redis中，所有的带抓取的对象和去重的指纹都存在所有的服务器公用的redis中
  • 所有的服务器共用一个redis中的request对象
  • 所有的request对象存入redis前，都会在同一个redis中进行判断，之前是否已经存入过
  • 在默认情况下所有的数据会保存在redis中

IP代理池

使用分布式爬虫后，就可以在单位时间内发起更多的请求。这种方式能够有效地应对访问频率限制，但经济成本很高。除了增加机器外，还可以使用IP切换的方式提高访问频率。假如用一台机器作为代理，轮流使用本机IP和代理IP发起请求，就能够将请访问频率提高1倍，9个代理能够将访问频率提升9倍。想要在1台机器上提高访问频率，可以使用多个IP代理。IP代理其实是维护一个IP池，爬虫程序每次发出请求时都从IP池中取出1个IP作为代理。

要注意的是，IP池中的IP地址需要由真实的机器（通常是服务器）提供代理服务，我们将这些提供代理服务的机器的IP地址收集起来，汇聚成一个“池”，所以叫作IP池。可以自己搭建用于IP代理的服务器，也可以直接从提供代理服务的商家购买IP。

浏览器指纹

浏览器指纹技术，作为一种用于确定客户端身份的手段，不仅依赖于IP地址，还能通过多种客户端特征信息生成“唯一”标识符。常见的客户端特征包括硬件信息、浏览器信息以及一些不可重复的信息（如Cookie、Token、用户凭证等）。这些信息在综合分析后可以构成一个浏览器指纹，用来唯一标识用户，尤其在用户未登录或未设置Cookie的情况下，依然能在一定程度上追踪和识别用户。

浏览器指纹的组成：

浏览器指纹通常是由多个特征信息组成的，这些特征信息包括但不限于以下几种：

类别 组成信息 详细描述
硬件信息 屏幕分辨率、色深、显示器的尺寸和DPI 描述显示设备的分辨率、色彩深度、显示器大小及DPI设置。
CPU核心数、处理器型号和类型 包括处理器的核心数量、型号、架构（如Intel、AMD）等信息。
系统的语言、操作系统的类型和版本号 包括操作系统（如Windows、macOS、Linux等）的语言、版本号等。
时区信息（用户当前时间与UTC的偏差） 用户所在时区与UTC的差异，反映用户的地理位置线索。
硬件设备的详细信息（如显卡、硬盘型号等） 硬件组件的特定信息，如显卡型号、硬盘型号、内存配置等。
浏览器信息 浏览器的 User-Agent 字符串 包含浏览器类型、操作系统、浏览器版本等信息。
浏览器支持的插件列表、字体、字体大小等 包括浏览器支持的插件、已安装的字体及其大小等。
浏览器的 navigator 对象中提供的信息（平台、语言、浏览器版本等） navigator 对象的各类信息，如浏览器平台、语言、浏览器类型与版本等。
不可重复信息 Cookie 和 Token 登录时由网站生成，用于用户身份验证。即使删除Cookie，仍有可能被追踪。
IP 地址 虽然可以通过代理或VPN更改，但仍是常见的标识信息，提供用户地理位置。

问题与挑战：

虽然浏览器指纹技术能提供一个相对唯一的标识符，但仍然存在一些问题，尤其是在某些环境中，设备特征可能高度相似。例如：

  • 同一硬件环境：如在某个网咖、学校或公司内，不同设备可能使用相同的硬件配置（CPU型号、显示器分辨率等），因此浏览器指纹的重复概率可能较高。
  • 同一网络环境：如果多台设备位于同一个网段或使用相同的公共Wi-Fi网络，它们的IP地址相同，因此产生的指纹也有可能相似或重复。

为了提高浏览器指纹的唯一性和难以伪造性，技术发展者开始采用一些新的技术和方法。

新技术：UUID、Canvas 和 WebGL

UUID

  • UUID（通用唯一识别码） 是一种标准化的标识符，它通过一系列算法生成独特的ID，理论上每个UUID都是唯一的。UUID有不同的版本，如基于时间、基于名称、基于随机数等方式生成。UUID的生成机制相当复杂，能确保在极高的概率下避免重复。
  • 在浏览器指纹中，UUID通常用于生成一个基于设备或用户会话的唯一标识符，它与其他客户端信息（如硬件配置、浏览器信息等）一起使用，可以进一步提高指纹的唯一性。

Canvas

  • Canvas指纹 是利用HTML5中的Canvas元素生成的，它通过绘制图形并将图形编码为Base64字符串，生成一个相对独特的字符串。Canvas指纹可以基于浏览器对图形的渲染方式、颜色深度、抗锯齿能力等特性生成，这些特性在不同的浏览器和操作系统中有所不同。
  • 由于Canvas指纹的生成不依赖于Cookie，它能够绕过浏览器禁用Cookie的限制，即使用户清除了Cookie，也不会影响指纹的生成。Canvas指纹是非常难以伪造的，除非两台设备完全相同，并且使用相同的浏览器和操作系统。

WebGL

  • WebGL指纹 通过浏览器中的WebGL（Web Graphics Library）渲染技术生成。WebGL允许网页直接使用GPU进行图形渲染，因此每个浏览器在渲染相同的图形时，可能会因硬件和驱动程序的不同而产生细微的差异。这些差异可以用来生成一个独特的指纹。
  • WebGL指纹比Canvas指纹更加依赖于设备的GPU和浏览器的渲染实现，因此在不同设备之间差异较大。在硬件配置、操作系统以及显卡驱动不同的情况下，WebGL指纹往往能够提供非常独特的标识符。

浏览器指纹的应用与反应：

  • 反爬虫技术：浏览器指纹广泛应用于反爬虫技术中，用于区分正常用户与自动化脚本或爬虫。通过收集浏览器指纹信息，网站能够检测并阻止大量的自动化请求，尤其是通过频繁修改IP地址的爬虫。
  • 广告追踪与个性化推荐：广告商和数据分析公司常常利用浏览器指纹来跟踪用户行为，进行精准广告投放，或提供个性化推荐。即使用户删除了Cookie或更换了设备，只要用户的浏览器指纹未发生变化，依然能够被跟踪。
  • 身份验证与安全防护：在安全领域，浏览器指纹可以用于增强身份验证，防止账号被盗用或进行恶意登录。通过验证浏览器指纹，服务商能够确认用户是否是之前登录的设备，从而提供额外的保护层。
