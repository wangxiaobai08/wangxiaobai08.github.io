使用intent在Activity穿梭

Intent 是 Android 中用于在组件（如 Activity、Service、BroadcastReceiver）之间传递消息和数据的对象。它是 Android 应用程序中实现组件通信的核心机制之一。Intent 可以用于启动 Activity、启动 Service、发送广播等。

Intent 的主要作用包括：

  1. 启动 Activity：通过 Intent 启动一个新的 Activity 或跳转到已有的 Activity。
  2. 启动 Service：通过 Intent 启动或绑定一个 Service。
  3. 发送广播：通过 Intent 发送广播消息，通知系统或其他应用。
  4. 传递数据：在组件之间传递数据（如字符串、整数、对象等）

启动Activity分为两种，显式调用和隐式调用。显式调用需要明确地指定被启动对象的组件信息，包括包名和类名，而隐式调用则不需要明确指定组件信息。原则上一个Intent不应该既是显式调用又是隐式调用，如果二者共存的话以显式调用为主。

显式调用 vs 隐式调用

调用方式 特点 示例
显式调用 明确指定目标组件的包名和类名。 Intent intent = new Intent(this, TargetActivity.class);
隐式调用 不指定目标组件，而是通过 Intent 的 Action、Category、Data 等信息匹配目标组件。 Intent intent = new Intent(Intent.ACTION_VIEW, Uri.parse("http://example.com"));

隐式调用的匹配流程

  1. 系统查找匹配的组件：
    • 当调用 startActivity(intent) 时，系统会查找所有声明了 IntentFilter 的组件。
    • 系统会将 Intent 的 Action、Category、Data 与每个组件的 IntentFilter 进行匹配。
  2. 匹配结果：
    • 如果找到唯一匹配的组件，则直接启动该组件。
    • 如果找到多个匹配的组件，则会弹出选择器（Chooser）让用户选择。
    • 如果没有找到匹配的组件，则会抛出 ActivityNotFoundException。

IntentFilter 是目标组件（如 Activity）在 AndroidManifest.xml 中声明的过滤规则，用于匹配隐式调用的 Intent。它包含以下三个主要部分：

  1. **<action>**：
    • 表示目标组件能够处理的动作。
    • 例如：android.intent.action.VIEW、android.intent.action.SEND。
    • 匹配规则：Intent 中的 Action 必须与 IntentFilter 中的至少一个 Action 匹配。
  2. **<category>**：
    • 表示目标组件的类别。
    • 例如：android.intent.category.DEFAULT、android.intent.category.BROWSABLE。
    • 匹配规则：
      • Intent 中的所有 Category 必须与 IntentFilter 中的 Category 完全匹配。
      • 如果 Intent 中没有指定 Category，系统会默认添加 android.intent.category.DEFAULT。
  3. **<data>**：
    • 表示目标组件能够处理的数据类型和格式。
    • 包括 scheme、host、port、path、mimeType 等。
    • 匹配规则：
      • Intent 中的 Data 必须与 IntentFilter 中的 Data 规则匹配。
      • 如果 IntentFilter 中定义了 Data 规则，则 Intent 必须包含 Data。

Activity的生命周期

原理性理解

进程间通信

APP从创建到销毁的生命周期里，必定同android系统有着千丝万缕的关系(比如：系统进程中的ActivityManagerService)，比如：电话来了,手机播放的音乐界面是否需要关闭？

这里主要涉及到两个进程：

system_server

system_server 进程是 Android 系统中的一个核心进程，负责运行 Java Framework 层的系统服务。它是 Android 系统启动时由 Zygote 进程 fork 出来的第一个进程。

主要功能是承载和管理大量的系统服务，例如：

  • **ActivityManagerService (AMS)**：管理 Activity 的生命周期、任务栈等。
  • **ApplicationThreadProxy (ATP)**：作为 AMS 与应用程序进程之间的桥梁，用于跨进程通信。
  • 其他服务：如 PackageManagerService、WindowManagerService 等。

system_server 进程中运行多个线程，其中许多是 Binder 线程。

App 进程

App 进程是应用程序的运行环境，每个 Android 应用通常运行在独立的进程中。进程由 Zygote 进程 fork 出来，并加载应用程序的代码和资源。负责运行应用程序的组件（如 Activity、Service、BroadcastReceiver 等）。

线程模型

  • 主线程（UI 线程）：
    • 负责处理 Activity/Service 等组件的生命周期方法。
    • 负责 UI 的绘制和事件处理。
  • Binder 线程：
    • 每个 App 进程至少有两个 Binder 线程：
      • **ApplicationThread (AT)**：用于与 system_server 进程通信，接收 AMS 的指令。
      • **ActivityManagerProxy (AMP)**：用于向 AMS 发送请求。
    • Binder 线程的创建和销毁由 Binder 驱动管理。
  • 其他线程：
    • 例如 Signal Catcher 线程：用于捕获和处理信号（如 ANR 信号）。

Binder用于不同进程之间通信,而handler用于同一个进程中不同线程的通信。见：

以 onPause() 为例：

1. 线程1：AMS 线程

  • AMS（ActivityManagerService） 是运行在 system_server 进程中的系统服务，负责管理 Activity 的生命周期。
  • 当需要暂停某个 Activity 时，AMS 会调用 ATP（ApplicationThreadProxy） 来发送暂停事件。
    • ATP 是 AMS 与 App 进程之间的桥梁，运行在 system_server 进程的另一个线程中。
    • 由于 AMS 和 ATP 在同一个进程（system_server）中，线程间可以直接调用，但需要注意多线程并发问题。

2. 线程2：ATP 线程

  • ATP 接收到 AMS 的暂停事件后，通过 Binder 机制 将事件传输到目标 App 进程。
    • Binder 是 Android 中用于跨进程通信的核心机制。
    • ATP 作为 Binder 客户端，将事件发送到 App 进程的 AT（ApplicationThread）。

3. 线程4：App 进程的 Binder 线程

  • AT（ApplicationThread） 是运行在 App 进程中的 Binder 线程，负责接收来自 system_server 进程的指令。
  • 当 AT 接收到暂停事件后，通过 Handler 消息机制 将事件发送到主线程（UI 线程）。
    • AT 将暂停事件封装为一个消息（例如 PAUSE_ACTIVITY），并通过 Handler.sendMessage() 发送到主线程的消息队列中。

4. 主线程：UI 线程

  • 主线程通过 Looper 循环遍历消息队列。
    • Looper.loop() 是主线程的核心逻辑，负责不断从消息队列中取出消息并处理。
  • 当主线程收到 PAUSE_ACTIVITY 消息时，会将消息分发给 ActivityThread.H.handleMessage() 方法。
    • ActivityThread.H 是主线程的 Handler，负责处理系统发送的生命周期事件。
  • 在 handleMessage() 方法中，经过层层调用，最终会调用到目标 Activity 的 onPause() 方法。

这便是由AMS完成了onPause()控制，那么同理Activity的其他生命周期也是这么个流程来进行控制的。

Activity的生命周期，都是其他线程通过handler发送消息给主线程，那么主线程中的ActivityThread的内部类H控制整个核心消息处理机制，通过H.handleMessage()来控制Activity的生命周期，在H类中共定义了50种消息。

主线程每到收到其他线程发送过来的不同的Handler消息，则都会触发相应的H.handleMessage.

典型情况下的生命周期分析

指在有用户参与的情况下，Activity所经过的生命周期的改变

在正常情况下，Activity会经历如下生命周期。

生命周期方法 调用时机 典型用途
onCreate() Activity 正在被创建时调用。 初始化 Activity，例如加载布局 (setContentView)、初始化数据等。
onStart() Activity 正在启动，即将进入前台时调用。这个时候其实可以理解为Activity已经显示出来了，但是我们还看不到。 执行需要在 Activity 可见时初始化的操作。
onResume() Activity 进入前台并可与用户交互时调用。onStart的时候Activity还在后台，onResume的时候Activity才显示到前台。 启动动画、注册监听器、恢复暂停的操作等。
onPause() Activity 失去焦点并进入暂停状态时调用。表示Activity正在停止，正常情况下，紧接着onStop就会被调用。在特殊情况下，如果这个时候快速地再回到当前Activity，那么onResume会被调用。 保存临时数据、停止动画、释放系统资源等。
onStop() Activity 完全不可见时调用。 释放占用资源、保存数据等。
onRestart() Activity 从停止状态重新启动时调用（在 onStop() 之后，onStart() 之前）。 执行 Activity 重新可见时需要恢复的操作。
onDestroy() Activity 正在被销毁时调用。 释放所有资源、注销监听器、清理内存等。

  • 针对一个特定的Activity，第一次启动，回调如下：onCreate -> onStart -> onResume。
  • 当用户打开新的Activity或者切换到桌面的时候，回调如下：onPause -> onStop。这里有一种特殊情况，如果新Activity采用了透明主题，那么当前Activity不会回调onStop。
  • 当用户再次回到原Activity时，回调如下：onRestart ->onStart -> onResume。
  • 当用户按back键回退时，回调如下：onPause -> onStop -> onDestroy。
  • 当Activity被系统回收后再次打开，生命周期方法回调过程和（1）一样，注意只是生命周期方法一样，不代表所有过程都一样，这个问题在下一节会详细说明。
  • 从整个生命周期来说，onCreate和onDestroy是配对的，分别标识着Activity的创建和销毁，并且只可能有一次调用。从Activity是否可见来说，onStart和onStop是配对的，随着用户的操作或者设备屏幕的点亮和熄灭，这两个方法可能被调用多次；从Activity是否在前台来说，onResume和onPause是配对的，随着用户操作或者设备屏幕的点亮和熄灭，这两个方法可能被调用多次。

Activity的生命周期中只有在以下3种状态之一，才能较长时间内保持状态不变。

  • Resumed（运行状态）：Activity处于前台，且用户可以与其交互。
  • Paused（暂停状态）: Activity被在前台中处于半透明状态或者未覆盖全屏的其他Activity部分遮挡。 暂停的Activity不会接收用户输入，也无法执行任何代码。
  • Stopped（停止状态）：Activity被完全隐藏，且对用户不可见；被视为后台Activity。 停止的Activity实例及其诸如成员变量等所有状态信息将保留，但它无法执行任何代码。

除此之外，其他状态都是过渡状态(或称为暂时状态)，比如onCreate()，onStart()后很快就会调用onResume()方法。

问题1:onStart和onResume、onPause和onStop从描述上来看差不多，对我们来说有什么实质的不同呢？

Android 提供这两对回调是为了更精细地控制 Activity 的生命周期，区分可见性和前台状态。onStart和onStop是从Activity是否可见这个角度来回调的，而onResume和onPause是从Activity是否位于前台这个角度来回调的，除了这种区别，在实际使用中没有其他明显区别。

问题2：假设当前Activity为A，如果这时用户打开一个新Activity B，那么B的onResume和A的onPause哪个先执行呢？

总结以下整体流程如下：

  1. 用户启动新 Activity B：
    启动请求会先由 Instrumentation 处理，然后通过 Binder 向 ActivityManagerService (AMS) 发送请求。

  2. AMS 处理请求：
    AMS 内部维护一个 ActivityStack，负责管理 Activity 的生命周期和状态。
    在 resumeTopActivityInnerLocked 方法中，AMS 会先检查当前栈顶的 Activity A 是否需要暂停。

  3. 暂停当前 Activity A：
    如果 Activity A 处于 Resumed 状态，AMS 会调用 startPausingLocked 方法，触发 Activity A 的 onPause 方法。

  4. 启动新 Activity B：
    在 Activity A 的 onPause 执行完成后，AMS 会通过 ActivityThread 的 ApplicationThread 调用 scheduleLaunchActivity 方法，启动新 Activity B。

  5. 新 Activity B 的生命周期调用：
    scheduleLaunchActivity 最终会调用 handleLaunchActivity 方法，依次执行 Activity B 的 onCreate、onStart 和 onResume。

  6. 顺序总结：

    • Activity A 的 onPause 先执行。
    • Activity B 的 onResume 后执行。

结论就是：当用户打开一个新 Activity B 时，当前 Activity A 的 onPause 会先执行，然后新 Activity B 的 onResume 才会执行。

异常情况下的生命周期分析

Activity被系统回收或者由于当前设备的Configuration发生改变从而导致Activity被销毁重建

情况1：资源相关的系统配置发生改变导致Activity被杀死并重新创建

当屏幕旋转等配置改变时，Activity 会被销毁并重建，系统会自动保存和恢复 View 的状态。可以通过 configChanges 阻止重建，避免不必要的生命周期调用。

  1. 系统配置改变的影响：
    当系统配置发生改变（例如屏幕旋转、语言切换等），默认情况下 Activity 会被销毁并重新创建。


    原因：系统需要根据新的配置重新加载资源（例如不同的布局、图片等)。

  2. Activity 的生命周期调用：

    • 销毁时：调用 onPause、onStop、onDestroy。

    • 保存状态：在 onStop 之前调用 onSaveInstanceState，保存 Activity 的状态数据（例如用户输入、滚动位置等）。

    • 重新创建时：调用 onCreate、onStart、onRestoreInstanceState，恢复之前保存的状态数据。

      当Activity在异常情况下需要重新创建时，系统会默认为我们保存当前Activity的视图结构，并且在Activity重启后为我们恢复这些数据，比如文本框中用户输入的数据、ListView滚动的位置等，这些View相关的状态系统都能够默认为我们恢复

  3. onSaveInstanceState 的调用时机：

    • 只在 Activity 异常终止时调用（例如配置改变导致的销毁）。
    • 正常销毁（例如用户按返回键）不会调用。

  4. View 的状态保存与恢复：

    • 系统会自动保存和恢复 View 的状态（例如文本框内容、列表滚动位置等）。
    • 通过 View 的 onSaveInstanceState 和 onRestoreInstanceState 方法实现。
    • 保存过程：Activity → Window → 顶层容器（如 DecorView） → 子 View，是一种典型的委托机制。

  5. 阻止 Activity 重新创建：
    可以通过在 AndroidManifest.xml 中为 Activity 指定 configChanges 属性，避免系统配置改变时重建 Activity。

情况2：资源内存不足导致低优先级的Activity被杀死

Activity 按照优先级从高到低可以分为以下三种：

  1. 前台 Activity：

    正在与用户交互的 Activity，优先级最高。

  2. 可见但非前台 Activity：

    例如 Activity 中弹出了一个对话框，导致 Activity 可见但无法与用户直接交互，优先级次之。

  3. 后台 Activity：

    已经被暂停的 Activity（例如执行了 onStop），优先级最低。

当系统内存不足时，系统就会按照上述优先级去杀死目标Activity所在的进程，并在后续通过onSaveInstanceState和onRestoreInstanceState来存储和恢复数数据。如果一个进程中没有四大组件在执行，那么这个进程将很快被系统杀死，因此，一些后台工作不适合脱离四大组件而独自运行在后台中，这样进程很容易被杀死。比较好的方法是将后台工作放入Service中从而保证进程有一定的优先级，这样就不会轻易地被系统杀死。

当系统配置发生改变（例如屏幕旋转、语言切换等），默认情况下 Activity 会被重新创建。

避免重建的方法：

可以通过在 AndroidManifest.xml 中为 Activity 指定 configChanges 属性，阻止系统重建 Activity。常用配置选项：

  • orientation：屏幕方向改变。
  • keyboardHidden：键盘可用性改变。
  • locale：语言或区域设置改变。

Acitivity的启动模式

Activity 的启动模式用于控制 Activity 实例的创建和任务栈的管理。默认情况下，每次启动一个 Activity 时，系统都会创建一个新的实例并将其放入任务栈中。这种默认行为在某些场景下可能会导致资源浪费（例如多次启动同一个 Activity 时重复创建实例）。为了解决这个问题，Android 提供了四种启动模式

任务栈：
任务栈Task，是一种用来放置Activity实例的容器，他是以栈的形式进行盛放，也就是所谓的先进后出，主要有2个基本操作：压栈和出栈，其所存放的Activity是不支持重新排序的，只能根据压栈和出栈操作更改Activity的顺序。

启动一个Application的时候，系统会为它默认创建一个对应的Task，用来放置根Activity。默认启动Activity会放在同一个Task中，新启动的Activity会被压入启动它的那个Activity的栈中，并且显示它。当用户按下回退键时，这个Activity就会被弹出栈，按下Home键回到桌面，再启动另一个应用，这时候之前那个Task就被移到后台，成为后台任务栈，而刚启动的那个Task就被调到前台，成为前台任务栈，手机页面显示的就是前台任务栈中的栈顶元素。

standard【标准模式】

默认模式

每次启动 Activity 时，都会创建一个新的实例，并将其放入任务栈中。

比如：

  • 任务栈状态：A -> B -> C
  • 启动 C 后：A -> B -> C -> C

这是一种典型的多实例实现，一个任务栈中可以有多个实例，每个实例也可以属于不同的任务栈。在这种模式下，谁启动了这个Activity，那么这个Activity就运行在启动它的那个Activity所在的栈中。比如Activity A启动了Activity B（B是标准模式），那么B就会进入到A所在的栈中。

非Activity类型的Context（如ApplicationContext）并没有所谓的任务栈，所以这就有问题了。解决这个问题的方法是为待启动Activity指定FLAG_ACTIVITY_NEW_TASK标记位，这样启动的时候就会为它创建一个新的任务栈

singleTop【栈顶复用模式】

如果目标 Activity 已经位于栈顶，则直接复用该实例，并调用 onNewIntent()。如果目标 Activity 不在栈顶，则创建新的实例。避免重复创建栈顶 Activity，例如通知点击跳转。

示例：

  • 任务栈状态：A -> B -> C
  • 启动 C 后：A -> B -> C（复用栈顶的 C）
  • 启动 B 后：A -> B -> C -> B（创建新的 B）

singleTask【栈内复用模式】

如果目标 Activity 已经存在于任务栈中，则直接复用该实例，并将其上方的所有 Activity 出栈，同时调用 onNewIntent()。如果目标 Activity 不存在于任务栈中，则创建新的实例。(需要保证某个 Activity 在栈中唯一性的场景，例如应用的主页。)

示例：

  • 任务栈状态：A -> B -> C
  • 启动 B 后：A -> B（复用 B，并将 C 出栈）

具体一点，当一个具有singleTask模式的Activity请求启动后，比如Activity A，系统首先会寻找是否存在A想要的任务栈，如果不存在，就重新创建一个任务栈，然后创建A的实例后把A放到栈中。如果存在A所需的任务栈，这时要看A是否在栈中有实例存在，如果有实例存在，那么系统就会把A调到栈顶并调用它的onNewIntent方法.

singleTask模式的Activity切换到栈顶会导致在它之上的栈内的Activity出栈。

这是一种单实例模式，在这种模式下，只要Activity在一个栈中存在，那么多次启动此Activity都不会重新创建实例

singleInstance【单实例模式】

目标 Activity 会单独运行在一个新的任务栈中，并且该任务栈中只能有一个 Activity。如果目标 Activity 已经存在，则直接复用该实例，并调用 onNewIntent()。(使用场景：需要完全独立运行的 Activity，例如拨号界面。)

示例：

  • 任务栈1：A -> B
  • 启动 C（singleInstance）后：
    • 任务栈1：A -> B
    • 任务栈2：C

这是一种加强的singleTask模式，它除了具有singleTask模式的所有特性外，还加强了一点，那就是具有此种模式的Activity只能单独地位于一个任务栈中，换句话说，比如Activity A是singleInstance模式，当A启动后，系统会为它创建一个新的任务栈，然后A独自在这个新的任务栈中，由于栈内复用的特性，后续的请求均不会创建新的Activity，除非这个独特的任务栈被系统销毁了。

什么是Activity所需要的任务栈呢？

TaskAffinity 是什么？

  • TaskAffinity 是一个标识符，表示 Activity 所需的任务栈名称。
  • 默认值：所有 Activity 的 TaskAffinity 默认为应用的包名。

与 singleTask 启动模式配对：

  • 当 Activity 的启动模式为 singleTask 时，系统会查找与 TaskAffinity 匹配的任务栈。
  • 如果找到匹配的任务栈，则在该栈中启动或复用 Activity；否则，创建新的任务栈。

与 allowTaskReparenting 属性配对：

  • 当 allowTaskReparenting 为 true 时，Activity 可以从一个任务栈转移到另一个任务栈。

通过 TaskAffinity 和启动模式，可以灵活控制 Activity 的任务栈行为。

1. 控制启动模式的 Flags

Flag 作用
FLAG_ACTIVITY_NEW_TASK 在一个新的任务栈中启动 Activity。如果目标 Activity 已经存在于某个任务栈中，则将该任务栈切换到前台。
FLAG_ACTIVITY_SINGLE_TOP 如果目标 Activity 已经位于任务栈的栈顶，则直接复用该实例，并调用 onNewIntent()。
FLAG_ACTIVITY_CLEAR_TOP 如果目标 Activity 已经存在于任务栈中，则清除其上方所有的 Activity，使其位于栈顶。
FLAG_ACTIVITY_REORDER_TO_FRONT 如果目标 Activity 已经存在于任务栈中，则将其移动到栈顶，而不重新创建实例。

2. 影响运行状态的 Flags

Flag 作用
FLAG_ACTIVITY_CLEAR_TASK 启动 Activity 时，清除与其关联的任务栈中的所有 Activity。
FLAG_ACTIVITY_EXCLUDE_FROM_RECENTS 启动的 Activity 不会出现在最近任务列表（Recent Tasks）中。
FLAG_ACTIVITY_NO_HISTORY 启动的 Activity 不会保留在任务栈中，用户离开后会被销毁。
FLAG_ACTIVITY_NO_ANIMATION 启动 Activity 时不使用过渡动画。
FLAG_ACTIVITY_MULTIPLE_TASK 与 FLAG_ACTIVITY_NEW_TASK 结合使用，强制创建一个新的任务栈，即使目标 Activity 已经存在。
